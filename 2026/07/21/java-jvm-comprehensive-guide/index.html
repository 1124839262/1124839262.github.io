一、JVM 概述

1.1 什么是 JVM

JVM（Java Virtual Machine，Java 虚拟机）是 Java 生态的核心，它是一台抽象的计算机，负责执行 Java 字节码。Java 之所以能实现"一次编写，到处运行"，靠的就是 JVM——开发者编译出的 .class 字节码文件，可以在任何安装了 JVM 的平台上运行。

1.2 JVM 整体架构

┌─────────────────────────────────────────────────┐ │ Java 源代码 (.java) │ └────────────────┬────────────────────────────────┘ │ javac 编译 ▼ ┌─────────────────────────────────────────────────┐ │ 字节码 (.class) │ └────────────────┬────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────┐ │ JVM 运行时数据区域 │ │ ┌──────────┐ ┌──────────┐ ┌──────────────┐ │ │ │ 堆 │ │ 方法区 │ │ 程序计数器 │ │ │ │ (Heap) │ │(Method) │ │ (PC Register)│ │ │ └──────────┘ └──────────┘ └──────────────┘ │ │ ┌──────────┐ ┌──────────┐ │ │ │ 虚拟机栈 │ │ 本地方法栈 │ │ │ │(VM Stack)│ │(Native) │ │ │ └──────────┘ └──────────┘ │ └─────────────────────────────────────────────────┘

二、JVM 内存模型

2.1 运行时数据区域

JVM 将内存划分为以下几个区域：

区域线程共享存储内容异常
堆（Heap）是对象实例、数组OutOfMemoryError
方法区（Method Area）是类信息、常量、静态变量、JIT 编译后的代码OutOfMemoryError
虚拟机栈（VM Stack）否局部变量、操作数栈、动态链接、方法出口StackOverflowError
本地方法栈（Native Stack）否本地方法（Native 方法）StackOverflowError
程序计数器（PC Register）否当前线程执行的字节码行号无

2.2 堆内存结构

堆是 GC 的主要管理区域，分为新生代和老年代：

┌──────────────────────────────────────────┐ │ 堆 Heap │ │ ┌─────────────────────┐ │ │ │ 新生代 Young │ Eden:S0:S1 │ │ │ ┌────┬─────┬─────┐ │ = 8:1:1 │ │ │ │Eden│ S0 │ S1 │ │ │ │ │ └────┴─────┴─────┘ │ │ │ └─────────────────────┘ │ │ ┌─────────────────────┐ │ │ │ 老年代 Old │ │ │ └─────────────────────┘ │ └──────────────────────────────────────────┘

  • 新生代（Young Generation）：新创建的对象先分配到这里，由 Minor GC 清理
    • Eden 区：大多数对象在这里诞生
    • Survivor 区（S0/S1）：经过一次 Minor GC 还存活的对象，会从 Eden 移到 Survivor 区
  • 老年代（Old Generation）：在 Survivor 区熬过多次 GC 的对象会晋升到老年代，由 Major GC / Full GC 清理

2.3 元空间（Metaspace）

JDK 8 之前叫永久代（PermGen），JDK 8 之后被元空间取代：

对比永久代（JDK 7及以前）元空间（JDK 8+）
存储位置JVM 堆内存中本地内存（Native Memory）
大小限制-XX:MaxPermSize-XX:MaxMetaspaceSize（默认无限制）
优势简单避免 OOM，自动扩容

三、类加载机制

3.1 类加载过程

一个类从加载到使用，经历以下 5 个阶段：

加载 → 验证 → 准备 → 解析 → 初始化

阶段说明
加载通过类的全限定名获取二进制字节流，转为方法区的运行时数据结构，生成 Class 对象
验证确保字节码符合 JVM 规范，包括文件格式验证、元数据验证、字节码验证、符号引用验证
准备为静态变量分配内存并赋默认值（int 为 �，oolean 为 alse，引用类型为
ull）
解析将符号引用替换为直接引用
初始化执行类构造器()，初始化静态变量和静态代码块

3.2 类加载器与双亲委派模型

mermaid graph TD A[Bootstrap ClassLoader] --> B[Extension ClassLoader] B --> C[Application ClassLoader] C --> D[自定义 ClassLoader]

类加载器加载范围
Bootstrap ClassLoader<JAVA_HOME>/lib 下的核心类库（rt.jar 等）
Extension ClassLoader<JAVA_HOME>/lib/ext 下的扩展类库
Application ClassLoader用户类路径（classpath）下的类

双亲委派模型：当一个类加载器收到类加载请求时，它不会自己先加载，而是将请求委派给父加载器，直到最顶层的 Bootstrap ClassLoader。如果父加载器找不到，才由子加载器自己加载。

这样做的好处：

  • 避免类的重复加载
  • 防止核心类库被篡改（比如自定义一个 java.lang.String 不会被加载）

四、垃圾回收（GC）

4.1 如何判断对象可回收

算法原理优缺点
引用计数法每个对象维护一个引用计数，为 0 时回收简单高效，但无法解决循环引用
可达性分析从 GC Roots 出发，沿着引用链搜索，不可达的对象可回收JVM 采用此方案，准确但需 STW

GC Roots 包括：

  • 虚拟机栈中引用的对象
  • 方法区中静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中引用的对象
  • 所有被同步锁持有的对象

4.2 垃圾回收算法

算法过程优点缺点
标记-清除标记存活对象 → 清除未标记对象简单内存碎片
标记-整理标记存活对象 → 将存活对象向一端移动 → 清理边界外无碎片移动成本高
复制将内存分为两块，每次只用一块，GC 时将存活对象复制到另一块无碎片，高效内存利用率低（50%）
分代收集新生代用复制算法，老年代用标记-整理/标记-清除综合效率高实现复杂

4.3 垃圾收集器

收集器作用区域算法特点适用场景
Serial新生代复制单线程，STW客户端模式
ParNew新生代复制Serial 的多线程版配合 CMS
Parallel Scavenge新生代复制关注吞吐量后台计算
Serial Old老年代标记-整理单线程客户端模式
Parallel Old老年代标记-整理多线程配合 Parallel Scavenge
CMS老年代标记-清除低延迟，并发收集B/S 服务端
G1全部标记-整理+复制可预测停顿，分 Region大内存服务端
ZGC全部染色指针超低延迟（<10ms）超低延迟场景

常用组合：

  • JDK 8 默认：Parallel Scavenge + Parallel Old
  • JDK 9+ 默认：G1
  • 低延迟场景：CMS / G1 / ZGC

4.4 GC 触发时机

GC 类型触发区域触发条件
Minor GC新生代Eden 区满
Major GC老年代老年代空间不足
Full GC整个堆 + 方法区System.gc()、老年代/方法区空间不足、担保失败

五、JVM 调优

5.1 常用 JVM 参数

`ash

堆内存

-Xms2048m # 初始堆大小
-Xmx2048m # 最大堆大小（建议与 -Xms 一致，避免动态扩容）
-Xmn512m # 新生代大小
-XX:NewRatio=2 # 老年代与新生代比值（默认 2，即老年代:新生代 = 2:1）
-XX:SurvivorRatio=8 # Eden 与 Survivor 比值（默认 8，即 Eden:S0 = 8:1）

元空间

-XX:MetaspaceSize=256m # 元空间初始大小
-XX:MaxMetaspaceSize=256m # 元空间最大大小

GC 日志

-XX:+PrintGCDetails # 打印 GC 详细日志
-XX:+PrintGCDateStamps # 打印 GC 时间戳
-Xloggc:gc.log # GC 日志文件路径

收集器选择

-XX:+UseSerialGC # Serial + Serial Old
-XX:+UseParallelGC # Parallel Scavenge + Parallel Old
-XX:+UseConcMarkSweepGC # ParNew + CMS
-XX:+UseG1GC # G1

OOM 时 dump

-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dump.hprof

面试题精选

1. 说一下 JVM 的内存模型？

JVM 内存分为线程共享和线程私有两部分。

线程共享：堆（存放对象实例和数组）和方法区（JDK 8 后为元空间，存放类信息、常量、静态变量）。

线程私有：虚拟机栈（每个方法执行时创建的栈帧，存储局部变量、操作数栈等）、本地方法栈（Native 方法服务）、程序计数器（记录当前线程执行位置）。

2. 堆的结构是怎样的？

堆分为新生代和老年代。新生代又分为 1 个 Eden 区和 2 个 Survivor 区（S0、S1），默认比例 8:1:1。新对象先在 Eden 区分配，经历 Minor GC 后存活的对象进入 Survivor 区，在 Survivor 区来回多次后进入老年代。大对象可能直接进入老年代。

3. 什么是双亲委派模型？为什么要这样设计？

双亲委派模型指：类加载器收到加载请求时，先委托父加载器去加载，父加载器找不到才由自己加载。

设计原因：一是避免重复加载，同一个类不会被不同加载器加载多次；二是安全考虑，防止核心类库被篡改，比如自定义的 java.lang.String 不会被加载，保证核心 API 的安全性。

4. 如何判断一个对象是否可以被回收？

JVM 使用可达性分析算法。从 GC Roots 出发，沿着引用链向下搜索，如果某个对象到 GC Roots 没有任何引用链相连，则判定为可回收对象。

GC Roots 包括：虚拟机栈中引用的对象、静态属性引用的对象、常量引用的对象、本地方法栈引用的对象、被同步锁持有的对象。

5. 常见的 GC 算法有哪些？各有什么优缺点？

算法优点缺点
标记-清除简单产生内存碎片
标记-整理无碎片移动对象成本高
复制无碎片，效率高浪费一半内存
分代收集综合效率最优实现复杂

JVM 实际采用分代收集：新生代对象朝生夕死，用复制算法；老年代对象存活率高，用标记-清除或标记-整理。

6. CMS 和 G1 的区别是什么？

对比维度CMSG1
内存布局连续的新生代+老年代将堆划分为多个 Region
算法标记-清除标记-整理 + 复制
碎片会产生碎片，需要 Full GC不会产生碎片
停顿时间可设置，但不可预测可预测停顿时间模型
适用场景内存较小（<4G）大内存（>4G）

7. 什么是 Minor GC、Major GC 和 Full GC？

  • Minor GC：发生在新生代，Eden 区满时触发，频率高，速度快。
  • Major GC：发生在老年代，老年代空间不足时触发，通常伴随一次 Minor GC。
  • Full GC：清理整个堆和方法区，STW 时间长，应尽量避免。

8. 什么情况下会触发 Full GC？

  • 调用 System.gc()（建议但不保证立即执行）
  • 老年代空间不足
  • 方法区（元空间）空间不足
  • Minor GC 时，Survivor 区放不下，且老年代也放不下（担保失败）
  • CMS GC 时出现 Concurrent Mode Failure

9. 线上 OOM 怎么排查？

  1. 找日志：先看 GC 日志和应用日志，确认是堆溢出还是元空间溢出
  2. dump 文件：如果有 -XX:+HeapDumpOnOutOfMemoryError 配置，直接拿 dump 文件分析
  3. jmap 生成 dump：jmap -dump:format=b,file=heap.hprof
  4. MAT 分析：用 Eclipse MAT 打开 dump 文件，查看 Dominator Tree 和 Leak Suspects
  5. 定位大对象：找到占用内存最多的对象，分析引用链，定位代码位置

10. JVM 调优你一般怎么做？

  1. 选收集器：根据业务场景选合适的 GC（响应优先选 G1/CMS，吞吐优先选 Parallel）
  2. 设堆大小：-Xms 和 -Xmx 设为相同值，避免动态扩容带来的性能损耗
  3. 设新生代：根据对象生命周期调整 -Xmn 或 -XX:NewRatio
  4. 开 GC 日志：-XX:+PrintGCDetails -XX:+PrintGCDateStamps
  5. 分析 GC 日志：关注 GC 频率、停顿时间、回收效果，逐步调优
  6. 压测验证：通过压测工具验证调优效果，反复迭代

前言

在单机应用中，我们可以用 synchronized 或 ReentrantLock 来保证线程安全。但在分布式系统中，多个服务实例运行在不同的 JVM 中，这些本地锁就无能为力了——你需要一把分布式锁来协调不同进程之间的资源访问。

举个例子：电商系统中，多个服务实例同时处理订单，如果不用分布式锁，可能会出现"超卖"问题——两个请求同时读到库存为 1，都认为可以下单，最终卖出 2 件。

市面上分布式锁的实现方案有很多，本文将对 6 种主流方案 逐一介绍它们的原理、使用方式和适用场景：

一、Redisson：基于 Redis 的分布式锁全家桶

1.1 是什么

Redisson 是一个基于 Redis 的 Java 驻内存数据网格（In-Memory Data Grid），它不仅仅是一个 Redis 客户端，更提供了分布式锁、分布式集合、分布式对象等一系列高级功能。

在分布式锁方面，Redisson 实现了 java.util.concurrent.locks.Lock 接口，用起来和本地锁几乎一样。

三、ShedLock：分布式定时任务锁

3.1 是什么

ShedLock 是一个专门用于分布式定时任务的锁框架。它的定位非常明确：确保同一个定时任务在同一时刻只有一个实例在运行。

这不是一个通用的分布式锁，而是专门解决"多实例部署时定时任务重复执行"的问题。

3.2 核心机制

ShedLock 在任务执行前向存储层（数据库/Redis/ZooKeeper）写入一条记录，任务执行完毕后删除或更新记录。其他实例在尝试执行任务时，发现记录已存在则跳过。

3.4 适用场景

  • 多实例定时任务：如凌晨批量对账、定时清理过期数据
  • 不需要通用的分布式锁，只需要"定时任务不重复执行"
  • 不想引入 Redis/ZooKeeper 等额外中间件（用数据库即可）

五、Curator InterProcessMutex：基于 ZooKeeper 的分布式锁

5.1 是什么

Apache Curator 是 ZooKeeper 的 Java 客户端框架，InterProcessMutex 是它提供的分布式可重入锁实现。

ZooKeeper 基于 CP 模型（一致性 + 分区容错），天然适合做分布式锁。

5.2 核心机制：临时顺序节点

ZooKeeper 分布式锁的实现原理：

  1. 所有客户端在 /locks 下创建临时顺序节点
  2. 序号最小的节点持有锁
  3. 每个客户端 watch 前一个节点
  4. 前一个节点释放（客户端断开或主动删除），下一个节点获得锁
  5. 客户端崩溃时，临时节点自动删除，不会死锁

  • 需要锁释放的绝对可靠性（客户端崩溃不会留下死锁）
  • 项目已经使用了 ZooKeeper

选型建议

注意事项

  1. 锁的粒度要合理：不要锁整个方法，尽量锁到资源级别（如按订单 ID 加锁）
  2. 设置合理的超时时间：防止死锁，但也不能太短导致业务没执行完就释放
  3. 释放锁时做校验：确保是自己加的锁才释放，避免误删其他客户端的锁
  4. Redisson 的 Watchdog 只对未指定 leaseTime 的 lock 生效：如果指定了过期时间，Watchdog 不会续期
  5. 数据库锁要注意清理过期锁：否则表会越来越大，且可能导致死锁
  6. ZooKeeper 锁要注意 Session 超时：网络抖动可能导致临时节点被删除，锁意外释放

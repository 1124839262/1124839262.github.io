一、WMS 是什么

WMS（Warehouse Management System，仓储管理系统）是管理仓库日常运作的软件系统，覆盖入库、出库、库存、拣货、盘点、库位管理等全流程。

和 ERP 的区别：ERP 管"账"，WMS 管"物"。ERP 告诉你库存有 100 件，WMS 告诉你这 100 件分别在哪个货架、哪个库位、哪个批次、哪个托盘上。

二、WMS 核心功能模块

2.1 功能全景图

2.2 入库管理

入库流程：

ASN 预约 → 收货 → 质检 → 上架 → 库位登记

环节功能关键点
ASN 预约供应商提前通知送货对接 ERP 采购单，生成 ASN 单号
收货扫描/录入到货商品支持条码/PDA 扫描，校验数量
质检检查商品质量合格/不合格分流，不合格品入不良品库
上架将商品放到具体库位按上架策略自动推荐库位，PDA 确认
库位登记更新系统库存记录商品、数量、批次、库位、托盘号

2.3 出库管理

出库流程：

订单接收 → 波次 → 分配库存 → 拣货 → 复核 → 包装 → 发货

环节功能关键点
订单接收从 ERP/OMS 拉取出库订单支持批量导入、实时推送
波次管理将订单按规则合并成波次按快递、区域、优先级等分组
分配库存锁定库存，指定拣货库位支持 FIFO/FEFO/指定批次
拣货按拣货单去库位拿货PDA 扫描确认，防止拿错
复核核对拣货商品是否正确扫描条码，重量校验
包装打包贴单生成面单，记录包材消耗
发货交接给物流更新订单状态，回传 ERP

2.4 库存管理

这是 WMS 最核心的模块，也是最容易出错的。

功能说明
库存查询按商品、库位、批次、状态多维度查询
库存冻结/解冻锁定库存（质检不合格、临期等），阻止出库
库存转移商品从一个库位移到另一个库位
库存调整盘点差异后的人工调整，需审批
库存移动日志每一条库存变动都要记录，可追溯

2.5 库位管理

仓库的物理空间建模：

仓库（Warehouse） └── 库区（Zone）—— 如：常温区、冷藏区、危险品区 └── 货架（Rack）—— 如：A01、A02 └── 库位（Location）—— 如：A01-01-01

库位属性说明
库位编码唯一标识，如 A01-03-02
库位类型存储位、拣货位、暂存位、退货位
长宽高承重用于计算能否放得下
混放规则单一 SKU 还是可混放

2.6 策略管理

策略是 WMS 的"大脑"，决定商品放在哪、从哪拿：

策略类型说明常见规则
上架策略收货后放到哪个库位就近上架、ABC 分类、周转率
分配策略出库时从哪个库位拿货FIFO（先进先出）、FEFO（先到期先出）
补货策略何时从存储位补到拣货位最小库存触发、波次预补货
波次策略如何合并订单拣货按快递公司、按区域、按订单量

3.2 库存扣减的并发控制

这是 WMS 最经典的技术难题。多个出库单同时扣减同一库位的库存时，如何保证不超卖？

`java
// 方案一：数据库行锁（推荐）
@Transactional
public void allocateInventory(Long itemId, Long locationId, String lotNo, BigDecimal qty) {
// SELECT … FOR UPDATE 锁定该行
Inventory inventory = inventoryMapper.selectForUpdate(itemId, locationId, lotNo);
if (inventory.getAvailableQty().compareTo(qty) < 0) {
throw new BusinessException(“库存不足”);
}
// 扣减库存
inventory.setLockedQty(inventory.getLockedQty().add(qty));
inventory.setAvailableQty(inventory.getQty().subtract(inventory.getLockedQty()));
inventoryMapper.updateById(inventory);

// 记录流水
insertInventoryLog(itemId, locationId, lotNo, qty.negate(), "出库", orderNo);

}

六、注意事项与踩坑经验

6.1 库存一致性

  • 库存流水表只增不改，所有库存变动必须记录，保证可追溯
  • 库存扣减用数据库行锁（SELECT FOR UPDATE），避免超卖
  • 不要在应用层计算库存，让数据库承担——SUM(qty) GROUP BY item_id
  • 定期做库存对账：实物数量 vs 系统数量，差异自动生成调整单

6.2 批次管理

  • 如果没有批次需求，不要强行上批次——复杂度翻倍
  • 批次编码建议：日期 + 流水号 + 供应商代码，如 20260721-001-S001
  • 效期管理：定时任务扫描临期/过期商品，自动冻结或转移

6.3 条码与编码规范

  • 统一使用 GS1-128 或 Code 128 条码，一维码即可，不要搞花里胡哨的二维码
  • 库位编码统一格式：库区-货架-层-位，如 A-01-03-02
  • 托盘编码建议：LPN + 日期 + 流水号，如 LPN202607210001

6.4 性能优化

  • 库存查询加索引：(item_id, location_id, lot_no, status)
  • 库存流水表按月分表，避免单表过大
  • 报表查询走只读库，不要压主库
  • PDA 接口响应时间控制在 500ms 以内，否则现场操作员会骂

6.5 容错与异常处理

  • 入库单状态流转：待收货 → 收货中 → 已完成，允许从"收货中"退回"待收货"
  • 出库单状态流转：待分配 → 拣货中 → 已复核 → 已发货，每个环节可能回退
  • 异常场景：
    • 收货多收/少收 → 生成差异单，通知采购
    • 拣货拿错 → 允许撤销拣货，库存回退
    • 发货漏发 → 支持补发，记录补发原因
    • 盘点差异 → 生成差异单，审批后调整

6.6 权限设计

角色权限范围
仓库管理员全部功能
入库操作员收货、上架、质检
出库操作员拣货、复核、包装
盘点员库存查询、盘点
只读用户只能看，不能操作

建议按仓库 + 库区做数据隔离，不同仓库的操作员互看不到对方的数据。

6.7 上架策略怎么选

策略逻辑适用场景
就近上架放到离收货区最近的空库位通用
ABC 分类A 类（高频）放近处，C 类（低频）放远处拣货效率优先
固定库位每个 SKU 绑定固定库位商品种类少、数量大
随机库位有空位就放，系统记录商品种类多、数量少

维度关键点
核心模块入库、出库、库存、库位、策略、报表
数据库库存表 + 库存流水表，流水只增不改
并发SELECT FOR UPDATE 行锁防超卖
硬件PDA、条码打印机、电子秤、PLC
对接ERP、OMS、TMS、MES

面试题精选

1. WMS 和 ERP 的库存有什么区别？

ERP 管的是账面库存，只记录商品和数量。WMS 管的是物理库存，精确到每个库位、每个批次、每个托盘。

举例：ERP 告诉你"A 商品库存 100 件"，WMS 告诉你"A 商品在 A01-03-02 库位有 30 件（批次 20260721），在 B02-01-01 库位有 70 件（批次 20260720）"。

2. WMS 如何防止库存超卖？

核心方案：数据库行锁 SELECT FOR UPDATE。扣减库存时，先锁住该行，读当前库存，判断是否足够，再扣减，事务提交释放锁。

高并发场景可以用 Redis 分布式锁，但要注意 Redis 和数据库的一致性，一般建议以数据库为准。

3. 入库和出库的状态机怎么设计？

入库：待收货 → 收货中 → 待上架 → 已完成，允许从"收货中"退回"待收货"，从"已完成"不能回退。

出库：待分配 → 已分配 → 拣货中 → 已复核 → 已发货，允许从"拣货中"退回"已分配"，"已发货"之后不能回退。

4. 库存流水表为什么只增不改？

为了审计可追溯。任何库存变动都能查到：谁、什么时候、做了什么操作、变动前后数量是多少。如果直接改库存表，出了问题无法追溯。

5. 盘点怎么做？差异怎么处理？

  1. 创建盘点任务（全盘 / 抽盘 / 动盘）
  2. 冻结盘点范围的库位
  3. 盘点员用 PDA 逐个库位扫描，录入实物数量
  4. 系统自动比对差异
  5. 差异单走审批流程
  6. 审批通过后，系统自动调整库存并记录流水

6. 拣货策略有哪些？

策略说明适用场景
按单拣货一张订单一个人拣到底订单量小
波次拣货多张订单合并，按商品汇总拣货后再分播订单量大
分区拣货不同人负责不同库区，接力完成仓库大
边拣边分拣货同时按订单分播订单量中等

7. WMS 怎么对接硬件设备？

  • PDA：安卓系统，通过 HTTP 调用后端 API，或用 WebSocket 实时推送任务
  • 条码打印机：后端生成条码指令（ZPL 语言），通过 Socket 发给打印机
  • 电子秤：串口读取重量，实时校验拣货重量是否在误差范围内
  • PLC/输送线：通过 Modbus TCP 下发分拣指令

8. WMS 系统架构怎么设计？

前端（Vue/React） → Nginx → Spring Cloud Gateway │ ┌───────────────┼───────────────┐ │ │ │ ┌────▼───┐ ┌────▼───┐ ┌─────▼────┐ │ 入库服务│ │ 出库服务│ │ 库存服务 │ └────┬───┘ └────┬───┘ └─────┬────┘ │ │ │ └───────────────┼───────────────┘ │ ┌─────────▼─────────┐ │ MySQL │ │ (主库 + 只读副本) │ └───────────────────┘

  • 微服务拆分：入库、出库、库存、库位、策略各一个服务
  • 库存服务是核心，其他服务通过库存服务操作库存
  • 使用 MQ 处理异步任务（如：出库完成后通知 TMS）
  • 报表查询走只读库，避免影响主库性能

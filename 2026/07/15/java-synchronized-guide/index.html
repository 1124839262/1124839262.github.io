前言

在 Java 并发编程中，synchronized 关键字是最基础、最常用的同步机制。它既是面试中的高频考点，也是实际开发中解决线程安全问题的第一把利器。但很多人对它的理解停留在"加锁"这个层面，稍微深入一点就答不上来了。

本文将从用法、底层原理、锁升级机制、JVM 实现到常见的面试题，做一个全面且深入的解析。

一、synchronized 的三种用法

1.1 修饰普通方法（实例锁）

锁的是当前实例对象 this。同一个实例的多线程调用会互斥，不同实例之间互不影响。

1.2 修饰静态方法（类锁）

锁的是当前类的 Class 对象（SynchronizedDemo.class）。无论多少个实例，同一时刻只有一个线程能执行该静态方法。

1.3 修饰代码块

代码块可以指定任意对象作为锁，灵活性最高。常见用法：

锁对象作用范围
this当前实例，等同于修饰普通方法
Xxx.class类的 Class 对象，等同于修饰静态方法
自定义 Object细粒度控制，避免与外部锁冲突

1.4 实例锁和类锁互斥吗？

不互斥。 实例锁锁的是对象，类锁锁的是 Class 对象，两者是完全不同的锁，不会互相阻塞。

两个线程分别调用 instanceMethod() 和 classMethod()，可以同时执行，不会阻塞。

三、锁升级（膨胀）过程详解

JDK 1.6 之后，synchronized 引入了偏向锁、轻量级锁来优化性能，锁可以从低级逐步升级到高级，但不会降级。

3.1 偏向锁（Biased Locking）

核心思想：大多数情况下，锁不仅不存在竞争，而且总是由同一个线程多次获取。偏向锁会在 Mark Word 中记录线程 ID，以后该线程再次进入同步块时，只需检查线程 ID 是否匹配，无需 CAS 操作。

获取流程：

  1. 检查 Mark Word 的偏向锁标志是否为 1，锁标志位是否为 01
  2. 如果是偏向锁，检查 Mark Word 中记录的线程 ID 是否为自己
  3. 如果是，直接执行同步代码，无需任何同步操作
  4. 如果不是，通过 CAS 尝试将线程 ID 设置为自己
  5. CAS 成功则执行同步代码，失败则进入偏向锁撤销

偏向锁撤销：当另一个线程尝试获取偏向锁时，偏向锁会撤销，升级为轻量级锁。撤销需要在**全局安全点（Safe Point）**进行，会暂停拥有偏向锁的线程。

注意：JDK 15 开始默认禁用偏向锁（因为在高并发场景下，偏向锁的撤销成本较高）。

3.2 轻量级锁（Lightweight Locking）

核心思想：当锁是偏向锁时，被另一个线程访问，偏向锁就会升级为轻量级锁。轻量级锁通过**自旋（Spin）**来等待锁释放，避免了线程阻塞和唤醒的开销。

获取流程：

  1. 在当前线程的栈帧中创建**锁记录（Lock Record）**空间
  2. 将 Mark Word 复制到锁记录中（称为 Displaced Mark Word）
  3. 通过 CAS 将 Mark Word 更新为指向锁记录的指针
  4. CAS 成功则获取锁，失败则进入自旋

自旋：线程在获取轻量级锁失败后，不会立即阻塞，而是会循环尝试 CAS 获取锁。自旋次数在 JDK 1.6 之后是自适应的，由 JVM 根据历史情况动态决定。

轻量级锁解锁：通过 CAS 将 Displaced Mark Word 替换回 Mark Word，如果 CAS 失败说明有其他线程在竞争，升级为重量级锁。

3.3 重量级锁（Heavyweight Locking）

当轻量级锁的自旋达到一定次数（或有多于一个线程在竞争），轻量级锁会膨胀为重量级锁。

重量级锁依赖操作系统的 **Mutex Lock（互斥锁）**实现，会导致线程阻塞和上下文切换，开销较大。但线程不会消耗 CPU 空转。

3.4 锁升级总结

锁状态优点缺点适用场景
偏向锁几乎没有额外开销撤销成本高只有一个线程访问
轻量级锁避免线程阻塞消耗 CPU 自旋线程交替执行
重量级锁不消耗 CPU线程阻塞/唤醒开销大高并发竞争

四、synchronized 的特性

4.1 可重入性

synchronized 是可重入锁。同一个线程可以多次获取同一把锁，不会自己把自己锁死。

Monitor 中的 _recursions 字段记录了重入次数，每次重入 +1，退出时 -1，归零时释放锁。

4.2 不可中断性

线程在等待 synchronized 锁时，无法被中断。调用 Thread.interrupt() 不会让等待锁的线程抛出 InterruptedException。

4.3 非公平性

synchronized 是非公平锁。等待队列中的线程不一定按顺序获得锁，新来的线程可能与等待线程竞争。

4.4 自动释放

synchronized 的锁在同步代码执行完毕或抛出异常时会自动释放，这也是它比 Lock 更容易使用的原因之一。

七、使用建议与注意事项

  1. 尽量缩小同步范围：只锁住需要同步的代码，减少锁持有时间。
  2. 使用私有 final 对象作为锁：避免外部代码意外获取锁导致死锁。
  3. 避免在锁内调用外部方法：可能导致死锁或性能问题。
  4. 不要在 synchronized 中使用 String 常量作为锁：String 常量池可能导致不同代码意外共享同一把锁。
  5. 避免嵌套锁：多把锁嵌套使用容易导致死锁，务必保持一致的加锁顺序。
  6. 优先使用并发集合：如 ConcurrentHashMap，它们内部做了更好的并发优化。

八、总结

synchronized 在 JDK 1.6 之后经过大量优化（偏向锁、轻量级锁、自适应自旋、锁消除、锁粗化等），性能已经不再是瓶颈。在日常开发中，如果不需要 ReentrantLock 的高级特性（可中断、超时、多条件），synchronized 是更简洁、更安全的选择。

附：高频面试题与答案

Q1：synchronized 底层是如何实现的？

答：synchronized 依赖 JVM 的 Monitor 机制实现。

  • 对于代码块：字节码中使用 monitorenter 和 monitorexit 指令。monitorenter 尝试获取对象的 Monitor，获取成功则 _count+1；monitorexit 则 _count-1，归零时释放锁。
  • 对于方法：在方法表中设置 ACC_SYNCHRONIZED 标志，JVM 根据此标志隐式调用 monitorenter/monitorexit。

底层数据结构是 ObjectMonitor（C++ 实现），包含 _owner（持有锁的线程）、_EntryList（竞争队列）、_WaitSet（等待队列）等字段。

Q2：请描述 synchronized 的锁升级过程？

答：锁升级方向是单向的：偏向锁 → 轻量级锁 → 重量级锁，不会降级。

  1. 偏向锁：初始状态。当只有一个线程访问同步代码时，Mark Word 记录该线程 ID，后续该线程再进入直接放行，无需同步操作。
  2. 轻量级锁：当另一个线程竞争偏向锁时，偏向锁撤销升级为轻量级锁。线程通过 CAS 尝试获取锁，失败则自旋等待。
  3. 重量级锁：当自旋达到一定次数或竞争线程数超过阈值，升级为重量级锁。未获得锁的线程阻塞，进入操作系统的 Mutex 等待队列，不消耗 CPU。

Q3：synchronized 是可重入的吗？如何实现的？

答：是的，synchronized 是可重入锁。

实现原理：在 ObjectMonitor 中维护了 _recursions（重入次数）和 _owner（持有线程）字段。当线程尝试获取锁时，先检查 _owner 是否为自己，如果是则 _recursions+1 直接放行；每次退出同步块时 _recursions-1，直到归零才真正释放锁。

Q4：synchronized 和 ReentrantLock 有什么区别？

答：

维度synchronizedReentrantLock
层面JVM 关键字JDK API
释放自动手动（finally 中 unlock）
可中断不支持支持 lockInterruptibly()
超时不支持支持 tryLock(timeout)
公平非公平可选公平/非公平
条件wait/notify 单一多个 Condition
性能JDK 6+ 优化后差距小功能更丰富

Q5：为什么 wait() 和 notify() 必须在 synchronized 中调用？

答：因为 wait() 和 notify() 是基于 Monitor 的机制实现的。调用 wait() 时，线程会释放当前持有的 Monitor 并进入 WaitSet 等待；调用 notify() 时，从 WaitSet 中唤醒一个线程进入 EntryList 竞争锁。

如果不在 synchronized 中调用，说明当前线程没有持有 Monitor，会抛出 IllegalMonitorStateException。

Q6：synchronized 修饰静态方法和普通方法有什么区别？

答：

  • 修饰普通方法：锁的是当前实例对象（this），不同实例之间不互斥。
  • 修饰静态方法：锁的是类的 Class 对象（Xxx.class），全局只有一个 Class 对象，所有实例共享同一把锁。

两者锁的是不同对象，所以不会互斥。

Q7：两个线程同时访问同一个对象的两个不同的 synchronized 方法，会阻塞吗？

答：会阻塞。 因为普通 synchronized 方法锁的都是同一个实例对象 this。线程 A 进入方法 1 时持有 this 锁，线程 B 想进入方法 2 也需要获取 this 锁，所以会被阻塞。

Q8：死锁的四个必要条件是什么？如何避免？

答：

四个必要条件（缺一不可）：

  1. 互斥条件：资源同时只能被一个线程持有。
  2. 请求与保持：线程持有资源的同时请求新资源。
  3. 不可剥夺：线程持有的资源不能被强制剥夺。
  4. 循环等待：线程间形成头尾相接的循环等待关系。

避免方法：

  • 保证加锁顺序一致，避免循环等待。
  • 使用 tryLock 设置超时（ReentrantLock）。
  • 使用 jstack 等工具检测死锁。

Q9：synchronized 能保证可见性吗？

答：能。 synchronized 在 JMM（Java Memory Model）中规定：

  • 线程加锁前，必须从主内存重新读取共享变量的最新值。
  • 线程解锁前，必须将工作内存中的共享变量刷新回主内存。

这保证了 synchronized 块内的修改对其他获取同一把锁的线程是可见的。

Q10：什么是锁消除和锁粗化？

答：这是 JVM 对 synchronized 的两种优化：

  • 锁消除（Lock Elimination）：JIT 编译器通过逃逸分析，如果发现同步块中的锁对象不可能被其他线程访问（如局部变量），就会直接去掉同步操作。
  • 锁粗化（Lock Coarsening）：如果一系列连续操作反复对同一个对象加锁解锁（如在循环中），JVM 会将锁的范围扩大，合并为一次加锁解锁，减少同步开销。

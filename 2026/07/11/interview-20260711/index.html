前言

今天参加了一场 Java 后端开发岗位的面试，面试官围绕 Java 基础、MySQL、JVM 调优、Redis 缓存和消息队列五个方向进行了深入提问。本文将所有问题整理记录，并给出详细的参考答案，既是复盘总结，也供后续复习参考。

一、Java 基础相关

1. Map 都有哪些实现类？

Java 中 Map 接口的常见实现类如下：

实现类线程安全有序性允许 null 键允许 null 值底层结构
HashMap否无序是是数组+链表+红黑树
LinkedHashMap否插入顺序/访问顺序是是HashMap+双向链表
TreeMap否按 key 自然排序否是红黑树
Hashtable是（synchronized）无序否否数组+链表
ConcurrentHashMap是（CAS+synchronized）无序否否数组+链表+红黑树

此外还有 IdentityHashMap、WeakHashMap、EnumMap 等较少使用的实现类。

2. HashMap 底层数据结构是什么？

JDK 1.8 中 HashMap 的底层数据结构是 数组 + 链表 + 红黑树：

  • 数组：Node<K,V>[] table，默认初始容量 16，通过哈希函数确定元素存放的桶位置
  • 链表：哈希冲突时，相同桶位置的元素以链表形式串联（尾插法，JDK 1.7 为头插法）
  • 红黑树：当某个桶的链表长度超过 8 且数组总容量 ≥ 64 时，链表转换为红黑树，查找时间复杂度从 O(n) 降为 O(log n)；当红黑树节点数 ≤ 6 时，退化为链表

关键参数：

  • 默认初始容量：16（DEFAULT_INITIAL_CAPACITY）
  • 负载因子：0.75（DEFAULT_LOAD_FACTOR）
  • 树化阈值：8（TREEIFY_THRESHOLD）
  • 树退化阈值：6（UNTREEIFY_THRESHOLD）
  • 最小树化容量：64（MIN_TREEIFY_CAPACITY）

3. HashMap 是线程安全的吗？

HashMap 不是线程安全的。 在多线程环境下使用 HashMap 会出现各种问题：

  • JDK 1.7：多线程并发扩容时，头插法会导致链表成环，后续 get() 操作进入死循环
  • JDK 1.8：改为尾插法解决了成环问题，但仍存在数据覆盖、size 不准确等问题

线程安全的替代方案：

  • ConcurrentHashMap（推荐，性能好）
  • Collections.synchronizedMap(new HashMap<>())（全局锁，性能差）
  • Hashtable（已过时，不推荐）

4. HashMap 多线程并发操作会出现什么问题？

问题说明
数据覆盖两个线程同时 put，哈希到同一桶位，后写入的覆盖先写入的，导致数据丢失
size 不准确多线程同时修改 size，++size 不是原子操作，计数可能出错
扩容数据丢失多线程同时触发扩容，迁移过程中部分节点可能丢失
死循环（JDK 1.7）并发扩容时头插法导致链表成环，get() 死循环，CPU 飙升到 100%
ConcurrentModificationException迭代过程中其他线程修改结构，触发 fail-fast 机制抛异常

5. ConcurrentHashMap 如何保证线程安全？

JDK 1.7 —— 分段锁（Segment）

  • 将数据分为 16 个 Segment，每个 Segment 是一个小的 Hashtable
  • 不同 Segment 的操作互不影响，并发度默认为 16
  • 锁粒度：Segment 级别

JDK 1.8 —— CAS + synchronized

  • 取消 Segment 分段锁，直接在 Node 数组上操作
  • put 流程：
    1. 计算 hash 定位桶位置
    2. 如果桶为空，使用 CAS 尝试写入（无锁操作）
    3. 如果桶不为空，对头节点加 synchronized 锁后操作
    4. 如果正在扩容，协助迁移数据（helpTransfer）
  • 锁粒度：桶的头节点级别，比分段锁更细，并发度等于数组长度

其他保证线程安全的机制：

  • sizeCtl 变量通过 CAS 控制扩容
  • CounterCell 数组统计 size，避免竞争
  • volatile 修饰 Node 的 val 和 next，保证可见性

6. 是否用过线程池？

是的，线程池是 Java 并发编程的核心工具。通过 ThreadPoolExecutor 创建线程池，避免频繁创建销毁线程的开销，提供任务队列管理和拒绝策略。

7. 线程池的参数都有哪些？

ThreadPoolExecutor 的 7 个核心参数：

参数含义
corePoolSize核心线程数，即使空闲也不回收
maximumPoolSize最大线程数，核心+非核心的总上限
keepAliveTime非核心线程空闲存活时间
unit存活时间单位
workQueue任务等待队列，如 LinkedBlockingQueue、SynchronousQueue
threadFactory线程工厂，用于自定义线程名、是否守护线程等
handler拒绝策略，队列满且线程数达到最大时的处理方式

四种拒绝策略：

  • AbortPolicy（默认）：抛出 RejectedExecutionException
  • CallerRunsPolicy：由提交任务的线程自己执行
  • DiscardPolicy：直接丢弃，不抛异常
  • DiscardOldestPolicy：丢弃队列中最旧的任务，重新提交当前任务

8. 生产环境一般如何配置线程池参数？

生产环境配置原则：

  1. 核心线程数（corePoolSize）：

    • CPU 密集型任务：CPU 核心数 + 1
    • IO 密集型任务：CPU 核心数 * 2 或 CPU 核心数 / (1 - 阻塞系数)
    • 实际需通过压测确定

  2. 最大线程数（maximumPoolSize）：根据系统承载能力和任务类型设定，一般为核心线程数的 2 倍左右

  3. 队列容量：使用有界队列（如 LinkedBlockingQueue(500)），避免无界队列导致 OOM

  4. 自定义线程工厂：给线程设置有意义的名称，便于排查问题

  1. 拒绝策略：通常使用 CallerRunsPolicy，既不丢弃任务也不会抛异常，还能起到限流效果

  2. 动态调整：生产环境可通过配置中心（Nacos/Apollo）动态修改线程池参数，并配合监控（线程活跃数、队列堆积数、拒绝次数）实时调优

二、MySQL 数据库相关

1. 日常使用什么数据库？是否用过 MySQL？

日常主要使用 MySQL，也接触过 SQL Server 和 PostgreSQL。MySQL 是目前最流行的开源关系型数据库，在互联网公司中使用最为广泛。

2. MySQL 索引是什么数据结构？

MySQL InnoDB 引擎的索引使用 B+ 树（B+ Tree）作为底层数据结构。

B+ 树的特点：

  • 非叶子节点只存储索引键，不存储数据，使得每个节点能容纳更多键值，树更矮胖
  • 所有数据都存储在叶子节点
  • 叶子节点之间通过双向链表串联，支持高效的范围查询和顺序扫描
  • 树的高度通常为 3~4 层，千万级数据只需 3~4 次 IO

3. MySQL 使用 B+ 树做索引有什么优势？

与 B 树、Hash 索引、二叉树等相比，B+ 树的优势：

对比项B+ 树B 树Hash 索引二叉树
范围查询高效（叶子链表顺序扫描）需要中序遍历不支持需要中序遍历
等值查询O(log n)O(log n)O(1)O(log n)
排序支持（有序）部分支持不支持支持
磁盘 IO少（3~4 次）较多少多（树高）
节点容量大（非叶节点只存键）小（存键+数据）—小

核心优势总结：

  1. IO 次数少：非叶节点不存数据，扇出大，树矮，查询只需 3~4 次磁盘 IO
  2. 范围查询高效：叶子节点链表串联，找到起点后顺序扫描即可
  3. 查询稳定：所有数据都在叶子节点，每次查询路径长度相同

4. 做过哪些数据库查询优化？

索引层面：

  • 遵循最左前缀原则，避免索引失效
  • 使用覆盖索引减少回表
  • 避免在索引列上使用函数或隐式类型转换
  • 联合索引中区分度高的列放前面
  • 长字符串使用前缀索引

SQL 层面：

  • 避免 SELECT *，只查需要的字段
  • 小表驱动大表，优化 JOIN 顺序
  • 用 EXISTS 替代 IN 处理大数据集
  • 避免在 WHERE 子句中对索引列做运算
  • 分页查询优化：深分页用 WHERE id > lastId 替代 LIMIT offset, size

架构层面：

  • 读写分离，主库写、从库读
  • 分库分表（ShardingSphere），按业务或数据量拆分
  • 引入 Redis 缓存热点数据

排查工具：

  • EXPLAIN 分析执行计划，关注 type、key、rows、Extra
  • 慢查询日志定位慢 SQL
  • SHOW PROFILE 分析 SQL 执行各阶段耗时

5. 事务的四大特性是什么？

事务的四大特性简称 ACID：

特性含义实现机制
Atomicity（原子性）事务中的操作要么全部成功，要么全部回滚undo log（回滚日志）
Consistency（一致性）事务执行前后，数据从一个一致状态转换到另一个一致状态由原子性+隔离性+持久性共同保证
Isolation（隔离性）并发事务之间互不干扰锁机制 + MVCC
Durability（持久性）事务提交后，数据的修改永久保存redo log（重做日志）

MySQL InnoDB 默认隔离级别为 可重复读（REPEATABLE READ），通过 MVCC（多版本并发控制）实现快照读，通过间隙锁防止幻读。

三、JVM 调优相关

1. 是否做过 JVM 相关调优？

是的，在处理线上系统性能问题时做过 JVM 调优。主要包括：

  • 调整堆内存大小和各区域比例
  • 选择合适的垃圾回收器
  • 排查内存泄漏和频繁 Full GC 问题
  • 分析 GC 日志优化停顿时间

四、Redis 缓存相关

1. 是否用过 Redis？

是的，Redis 是项目中使用最频繁的中间件之一，主要用于缓存、分布式锁、限流计数、排行榜等场景。

2. Redis 有哪些数据结构？

Redis 有 5 种基本数据结构和 3 种扩展数据结构：

类型底层编码常见用途
Stringint / embstr / raw缓存、计数器、分布式锁
Listquicklist（ziplist + linkedlist）消息队列、文章列表
Hashziplist / hashtable对象存储（用户信息等）
Setintset / hashtable标签、共同好友、去重
ZSetziplist / skiplist+hashtable排行榜、延迟队列
HyperLogLog—基数统计（UV 计数）
Bitmap—签到、布隆过滤器
GeoZSet地理位置、附近的人

3. ZSet 底层是什么数据结构？

ZSet（有序集合）的底层实现有两种编码：

  1. ziplist（压缩列表）：当元素数量 ≤ 128 且每个元素长度 ≤ 64 字节时使用，内存紧凑，适合小数据量

  2. skiplist + hashtable（跳跃表 + 字典）：当不满足 ziplist 条件时使用

    • skiplist：支持范围查询，查找 O(log n)，按 score 有序
    • hashtable：保存 member → score 的映射，支持 O(1) 按 member 查找 score

4. Redis 有哪几种持久化方式？

方式原理优点缺点
RDB定时将内存数据快照写入磁盘（dump.rdb）文件紧凑、恢复速度快、对性能影响小可能丢失两次快照之间的数据
AOF将写命令追加到日志文件（appendonly.aof）数据更安全、最多丢 1 秒数据文件大、恢复慢、对性能有影响
混合持久化RDB+AOF 混合，重写时前半部分 RDB 格式，后半部分 AOF 格式兼顾恢复速度和数据安全需 Redis 4.0+

生产环境推荐：开启 AOF + 混合持久化，同时保留 RDB 作为备份。

AOF 文件增大的核心原因是只追加不删除，即使 key 被删除或覆盖，旧命令仍然保留。

优化方式：AOF 重写（rewrite）

触发方式：

  • 手动触发：BGREWRITEAOF 命令
  • 自动触发：配置 auto-aof-rewrite-percentage 和 auto-aof-rewrite-min-size

Redis 4.0+ 开启混合持久化后，重写时使用 RDB 格式写入前半部分，大幅减小文件体积。

6. 如何保证 Redis 缓存与数据库的数据一致性？

缓存与数据库的一致性是经典难题，常见策略：

Cache Aside Pattern（旁路缓存，最常用）：

  • 读：先读缓存，命中则返回；未命中则读数据库，写入缓存
  • 写：先更新数据库，再删除缓存

为什么是删除缓存而不是更新缓存？

  • 避免并发写导致缓存与数据库不一致
  • 删除更简单，下次读取时懒加载

为什么是先更新数据库再删除缓存？

  • 先删缓存再更新数据库：并发时其他线程可能读到旧数据并写入缓存
  • 先更新数据库再删缓存：极端并发下仍可能出现短暂不一致，但概率低且时间窗口极小

强一致性方案（按需选择）：

  • 延迟双删：先删缓存 → 更新数据库 → 延迟 N 毫秒 → 再删缓存
  • 监听 binlog：Canal 监听 MySQL binlog，异步删除/更新缓存
  • 分布式事务：Seata 等保证强一致（性能代价大）

五、消息队列相关

1. 是否用过消息中间件？

是的，项目中使用过 RocketMQ 和 RabbitMQ：

  • RocketMQ：用于订单异步处理、延迟消息（超时未支付自动取消）、事务消息等场景
  • RabbitMQ：用于日志收集、异步通知等轻量级场景

2. RocketMQ 是否出现过消息丢失的情况？

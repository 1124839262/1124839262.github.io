前言

在单体应用中，事务管理是一件相对简单的事情——Spring 的 @Transactional 注解就能搞定一切。但当你把单体应用拆分成微服务后，原本的一个本地事务变成了跨多个服务、多个数据库的分布式事务，问题就变得复杂了。

举个例子：电商下单流程中，订单服务需要创建订单，库存服务需要扣减库存，账户服务需要扣减余额。这三个操作分别在三台不同的服务器上、操作三个不同的数据库。如果库存扣减成功了，但账户扣减失败了，传统的事务机制就无法保证数据一致性了。

本文将带你从理论到实践，全面了解分布式事务的解决方案，并重点介绍 Seata 的使用场景和实战方式。

一、分布式事务基础理论

1.1 CAP 定理

在分布式系统中，CAP 定理指出一个分布式系统最多只能同时满足以下三个特性中的两个：

特性含义说明
Consistency（一致性）所有节点在同一时间看到的数据完全相同写操作后，所有读操作都能获取到最新值
Availability（可用性）每个请求都能获得非错误的响应系统一直可用，不会返回错误或超时
Partition Tolerance（分区容错性）系统在出现网络分区时仍能正常运作节点间通信失败不影响整体服务

由于网络分区不可避免（P 必须满足），因此分布式系统只能在 CP 和 AP 之间做取舍。分布式事务天然倾向于 CP，但也会根据场景做权衡。

1.2 BASE 理论

BASE 理论是对 CAP 中 AP 的延伸，它主张：

  • Basically Available（基本可用）：系统出现故障时允许损失部分可用性，但核心功能可用
  • Soft state（软状态）：允许系统中的数据存在中间状态，该状态不影响系统整体可用性
  • Eventually consistent（最终一致性）：系统中的所有数据副本经过一定时间后，最终能够达到一致的状态

BASE 理论是分布式事务中"柔性事务"的思想基础——不强求实时一致性，但保证最终一致性。

二、经典分布式事务解决方案

2.1 两阶段提交（2PC / XA）

2PC 是最经典的分布式事务协议，它引入一个**事务协调器（Coordinator）**来协调多个参与者的提交。

执行流程：

优点：

  • 原理简单，各大数据库都有支持（MySQL 从 5.5 开始支持 XA）
  • 尽量保证强一致性

缺点：

  • 单点问题：协调器宕机会导致所有参与者阻塞
  • 同步阻塞：第一阶段锁定资源后，必须等待第二阶段完成才能释放
  • 数据不一致风险：第二阶段如果部分参与者收到 Commit 而部分未收到，会导致数据不一致

2.2 TCC（Try-Confirm-Cancel）

TCC 是一种补偿型事务方案，它将事务拆分为三个阶段：

阶段含义操作
Try尝试执行业务完成业务检查，预留资源
Confirm确认提交真正执行业务，使用 Try 阶段预留的资源
Cancel取消回滚释放 Try 阶段预留的资源

优点：

  • 无锁，性能好
  • 不依赖数据库事务，可用于跨系统调用

缺点：

  • 业务侵入性强，需要为每个操作编写 Try/Confirm/Cancel 三个方法
  • 需要处理幂等、空回滚、悬挂等问题

2.3 本地消息表

本地消息表方案的核心思想是将分布式事务拆分为多个本地事务，通过消息表来保证最终一致性。

优点：

  • 实现简单，不依赖外部组件
  • 支持最终一致性

缺点：

  • 消息表与业务耦合
  • 需要额外的定时任务处理重试

2.4 MQ 事务消息

RocketMQ 在 4.3 版本后支持事务消息，本质上是将本地消息表逻辑内置到了 MQ 中。

优点：

  • 业务侵入性低
  • RocketMQ 原生支持，可靠性高

缺点：

  • 仅支持 RocketMQ
  • 需要实现回查接口

三、Seata：一站式分布式事务解决方案

3.1 Seata 是什么

Seata（Simple Extensible Autonomous Transaction Architecture）是阿里巴巴开源的一款分布式事务解决方案，提供了高性能、低侵入的分布式事务服务。其前身是阿里内部的 GTS（Global Transaction Service），2019 年开源并进入 Apache 孵化器，2023 年毕业成为 Apache 顶级项目。

3.2 Seata 核心架构

Seata 采用三层架构：

组件角色职责
TC（Transaction Coordinator）事务协调器独立部署的 Server，维护全局事务状态，协调各分支事务的提交或回滚
TM（Transaction Manager）事务管理器嵌入在业务应用中，负责定义全局事务边界，发起全局事务的开启、提交、回滚
RM（Resource Manager）资源管理器嵌入在业务应用中，负责管理分支事务涉及的资源，向 TC 注册分支事务并报告状态

3.3 Seata 四大事务模式

Seata 提供了四种事务模式，每种模式适用于不同的场景：

3.3.1 AT 模式（Automatic Transaction）

AT 模式是 Seata 的默认模式，也是使用最广泛的模式。它基于两阶段提交协议，但做了大量优化，核心是通过自动生成反向 SQL来实现自动回滚。

执行流程：

核心机制：

  • undo_log 表：每个业务库中都需要创建一张 undo_log 表，用于记录数据变更的前镜像和后镜像
  • 全局锁：Seata 在提交前会获取全局锁，防止其他事务修改同一数据
  • 读写隔离：默认读已提交，写操作通过全局锁实现隔离

优点：

  • 对业务代码零侵入，只需加 @GlobalTransactional 注解
  • 自动生成回滚日志，无需手动编写补偿逻辑

缺点：

  • 仅支持关系型数据库（通过 JDBC）
  • 需要为每个业务库创建 undo_log 表
  • 默认隔离级别为读未提交，高频并发场景需注意

3.3.2 TCC 模式

Seata 的 TCC 模式与经典 TCC 一致，但提供了框架层面的支持，帮助开发者处理幂等、空回滚、悬挂等问题。

Seata TCC 关键注解：

useTCCFence = true 是 Seata 提供的TCC 防悬挂机制，能自动处理：

  • 空回滚：Try 未执行时 Cancel 被调用，自动忽略
  • 悬挂：Cancel 先于 Try 到达，Try 被拒绝执行
  • 幂等：重复调用 Try/Confirm/Cancel 自动去重

优点：

  • 性能好，无锁
  • 不依赖数据库，可用于调用第三方接口

缺点：

  • 业务侵入性高，需要实现三个方法
  • 补偿逻辑复杂度高

3.3.3 Saga 模式

Saga 模式是一种长事务解决方案，它将一个长事务拆分为多个有序的本地事务，每个本地事务都有对应的补偿操作。

执行流程：

Saga 实现方式：

  • 编排式（Choreography）：每个服务监听事件并决定下一步操作
  • 控制式（Orchestration）：由 Saga 协调器统一调度（Seata 采用此方式）

Seata Saga 特点：

  • 支持 JSON 或 Java 定义状态机
  • 支持服务编排，适合复杂长流程
  • 异步执行，无锁，性能高

优点：

  • 无锁，适合长事务
  • 支持非数据库操作
  • 异步执行，吞吐量高

缺点：

  • 最终一致性，无法保证实时一致
  • 补偿逻辑复杂，需要仔细设计

3.3.4 XA 模式

Seata 的 XA 模式基于数据库的 XA 协议实现，追求强一致性。

执行流程：

优点：

  • 强一致性，数据不丢失
  • 数据库原生支持，可靠性高

缺点：

  • 需要数据库支持 XA 协议
  • 同步阻塞，性能较差
  • 占用数据库连接时间长

3.4 四种模式对比

维度ATTCCSagaXA
一致性最终一致最终一致最终一致强一致
性能中高高低
业务侵入性无高中无
回滚方式自动反向 SQL手动补偿手动补偿数据库自动回滚

四、Seata 模式选型指南

根据不同的业务场景，推荐以下选型策略：

典型场景推荐

场景推荐模式理由
电商下单（创建订单+扣库存+扣余额）AT都是数据库操作，事务短，零侵入
高并发秒杀TCC性能好，无锁，可自定义优化
跨系统调用（含第三方 API）TCCAT 无法处理非数据库操作
跨境支付/物流调度Saga流程长，容忍最终一致性
金融核心交易XA强一致性要求
老系统改造（无法改代码）AT零侵入，只需加注解
定时任务批量处理Saga允许长时间运行，异步执行

六、Seata 生产环境注意事项

6.1 高可用部署

  • Seata Server 至少部署 2 台，通过 Nacos 实现服务发现
  • 事务日志存储使用 MySQL 主从，避免单点
  • 配置中心和注册中心使用 Nacos 集群

6.2 性能优化建议

优化项建议
undo_log 定期清理配置定时任务清理过期的 undo_log 记录
合理设置事务超时默认 60 秒，长事务适当调大
避免大事务单个全局事务不要包含太多分支事务
数据库连接池适当增大连接池，避免连接等待
异步提交非核心链路可考虑异步化，减少事务范围

6.3 常见问题排查

问题可能原因解决方案
xid 未传递Feign 拦截器未配置确认 seata.enable-auto-data-source-proxy=true
undo_log 未生成数据源未代理检查 DataSource 是否为 Seata 代理
全局锁冲突并发操作同一数据拆分热点数据，或改用 TCC 模式
事务超时回滚业务执行时间过长调大超时时间，或优化业务逻辑

七、总结

本文从分布式事务的基础理论出发，介绍了 CAP 定理、BASE 理论，以及四种经典的分布式事务解决方案：2PC/XA、TCC、本地消息表、MQ 事务消息。然后重点介绍了 Seata 的 AT、TCC、Saga、XA 四种模式及其适用场景，并给出了完整的 Spring Boot 集成实战代码。

快速选型速查：

  • 大多数场景用 AT 模式，零侵入，开发效率最高
  • 高并发或涉及第三方接口用 TCC 模式
  • 长流程、跨系统用 Saga 模式
  • 金融等强一致性场景用 XA 模式

分布式事务没有银弹，每种方案都有其适用场景和局限性。在实际项目中，最重要的是根据业务需求选择最合适的方案，而不是追求最"高级"的技术。

秒杀系统概述

秒杀场景的核心特征：短时间、高并发、库存有限。典型场景如 618 大促、双11抢购、抢红包等。

核心挑战

挑战描述
超卖库存只有100件，卖出101件
高并发瞬时流量可达日常百倍甚至千倍
恶意请求脚本刷单、黄牛抢购
数据一致性缓存与数据库之间的数据同步
服务可用性流量洪峰下系统不能崩溃

整体架构

核心思路：层层拦截，异步削峰，最终一致

一、前端层面

1. 页面静态化 + CDN

  • 秒杀页面提前静态化，部署到 CDN
  • 动态数据（库存、倒计时）通过接口异步加载
  • 避免每次请求都打到应用服务器

2. 按钮防重复点击

3. 倒计时 + 随机URL

  • 前端倒计时到点后才暴露秒杀接口地址
  • 接口地址中加入随机token，防止提前猜测

二、网关与限流

1. Nginx 限流

2. 网关层限流（Sentinel / 自研）

  • 令牌桶算法：控制每秒放行请求数
  • 按用户ID维度限流：防止单用户高频请求
  • 按商品维度限流：库存100件，放行量不超过库存的3-5倍

三、核心：库存扣减

方案一：Redis 预扣库存（推荐）

方案二：Redis + Lua 脚本（更严谨）

将判断和扣减合并为一个原子操作，避免并发窗口问题：

  • 返回 1：扣减成功
  • 返回 -1：库存不足
  • 返回 -2：重复下单

四、异步下单

MQ 削峰

  • 秒杀服务只负责预扣库存 + 发消息，快速返回
  • 订单服务消费消息，创建订单，扣减数据库库存
  • 下游处理能力不足时，MQ 自然堆积，保护数据库

五、防超卖保障

数据库层兜底

即使 Redis 扣减成功，数据库仍需做最终校验：

stock > 0 条件确保数据库层不会超卖。影响行数为 0 则说明库存已耗尽，需要回补 Redis 库存。

六、防刷与安全

手段说明
验证码秒杀前先过图形/滑块验证码，拦截脚本
IP限流同一IP短时间请求次数上限
用户维度限流同一用户同一商品只能抢一次
接口隐藏秒杀URL动态生成，加上时效token
黑名单风控系统识别异常用户，加入黑名单
隐藏库存前端不展示实时库存，只显示有货/售罄

七、库存回滚

以下场景需要回补库存：

  1. 订单超时未支付 -> 关闭订单，库存回补
  2. 下单失败（DB异常）-> 回补 Redis 库存
  3. 用户取消订单 -> 库存回补

注意：回补库存后，需要判断是否需要重新开放购买（可能已标记售罄）。

八、数据一致性对账

机制说明
实时对账Redis库存 + DB已下单数 = 总库存
定时任务每分钟扫描，发现不一致则告警并修复
活动结束后清算Redis库存，以DB为准做最终校准

九、高可用保障

  • 服务多实例部署，Nginx 负载均衡
  • Redis 集群（哨兵或 Cluster），避免单点故障
  • MQ 集群，保证消息不丢失
  • 降级开关：库存售罄后，直接在网关层返回，请求不再打到后端
  • 熔断：下游服务异常时，快速失败，避免级联故障

总结：秒杀系统核心原则

  1. 尽量将请求拦截在上游 – 能在CDN返回的就不到Nginx，能在Nginx返回的就不到网关
  2. 缓存抗读，MQ抗写 – 读走Redis，写走MQ异步化
  3. 库存扣减原子化 – Redis Lua脚本或DECR原子操作
  4. 数据库做最终兜底 – stock > 0 条件防超卖
  5. 异步最终一致 – 接受短暂不一致，通过对账保证最终一致
  6. 可降级可熔断 – 极端情况下宁可少卖，不能超卖

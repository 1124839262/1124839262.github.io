为什么需要限流

在高并发场景下，系统资源（CPU、内存、连接数、数据库连接池）是有限的。如果没有限流保护，突发流量可能导致：

  • 服务雪崩：后端服务被压垮，影响所有用户
  • 数据库过载：大量请求同时打到 DB，导致慢查询堆积
  • 带宽浪费：恶意爬虫或异常流量消耗带宽资源

限流的核心目标：在保证服务质量的前提下，尽可能多地处理合法请求。

一、固定窗口计数器 (Fixed Window)

算法原理

将时间划分为固定的窗口（如 1 秒、1 分钟），每个窗口内独立计数。

OpenResty 实现

优缺点

优点缺点
实现简单，性能高临界问题：窗口边界处可能产生 2 倍突发
内存占用恒定无法应对突发流量平滑

临界问题示例：假设限制 100 次/分钟，在 00:59 发了 100 次，01:00 窗口重置又发了 100 次，1 秒内实际处理了 200 次。

二、滑动窗口计数器 (Sliding Window)

算法原理

不按固定边界切分，而是记录每个请求的时间戳，统计最近 N 毫秒/秒内的请求数量。

OpenResty 实现

优缺点

优点缺点
解决了临界问题内存占用随请求量增长
流量控制更精确需要存储每个请求的时间戳
支持动态调整窗口编解码 JSON 有一定 CPU 开销

三、漏桶算法 (Leaky Bucket)

算法原理

想象一个底部有孔的桶：

  • **请求（水）**从顶部注入桶中
  • 水以恒定速率从底部漏出（处理请求）
  • 如果桶满了，新请求被拒绝

核心特点：输出速率恒定，能够强制平滑流量。

OpenResty 实现

适用场景

场景说明
保护下游服务强制限制到后端的速率，防止打挂
流量整形将突发流量平滑为恒定输出
API 网关限流保证后端服务稳定

优缺点

优点缺点
输出速率恒定，适合保护后端无法处理突发流量（即使桶有空余）
实现相对简单需要精确计算漏水速率
内存占用固定参数调节需要经验

四、令牌桶算法 (Token Bucket)

算法原理

系统以恒定速率向桶中放入令牌：

  • 每个请求消耗 1 个令牌
  • 有令牌 → 允许通过
  • 无令牌 → 拒绝或等待
  • 桶满时令牌溢出丢弃

核心特点：允许一定程度的突发流量（桶中积累的令牌）。

OpenResty 实现

适用场景

场景说明
允许突发流量的API如用户快速连续点击按钮
CDN/边缘节点平衡用户体验和系统保护
微服务调用允许短时突发但控制平均速率

优缺点

优点缺点
允许合理突发突发后需要时间恢复
参数直观易懂长时间空闲后首次请求可能触发大突发
业界广泛采用(Google Guava, Envoy等)需要合理设置 rate 和 burst

维度固定窗口滑动窗口漏桶令牌桶
精度⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐

总结

算法核心思想选型建议
固定窗口时间片独立计数简单场景，对精度要求不高
滑动窗口动态窗口统计需要精确控制的场景
漏桶恒定速率输出保护后端服务，强制平滑
令牌桶恒定速率输入允许突发的 API 场景

生产环境建议：Nginx 内置模块做第一道防线 + Lua 自定义逻辑做精细化控制 + Redis 做分布式全局限流。

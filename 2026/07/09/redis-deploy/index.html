概述

Redis 在生产环境中通常不会以单节点方式运行，而是采用主从复制或**集群（Cluster）**模式来保证高可用与数据安全。本文将详细介绍两种主流部署方案：

  • 一主两从 + Sentinel：适用于读写分离、高可用场景
  • Redis Cluster：适用于数据量较大、需要水平扩展的场景

一、一主两从 + Sentinel 部署

1.1 架构说明

  • Master：负责读写操作
  • Slave：只读，从 Master 同步数据
  • Sentinel：监控主从状态，自动故障转移

二、Redis Cluster 部署

2.1 架构说明

Redis Cluster 将数据分为 16384 个槽（slot），分配到不同 Master 节点。每个 key 通过 CRC16(key) % 16384 计算所属槽位。

2.2 最小集群要求

  • 至少 6 个节点：3 Master + 3 Slave
  • 每个 Master 至少 1 个 Slave 保证高可用

三、两种方案对比

对比项一主两从 + SentinelRedis Cluster
数据分片不支持，所有数据在 Master支持，数据分布在多个 Master
水平扩展不支持，只能垂直扩展支持，添加节点重新分配槽位
高可用Sentinel 自动故障转移内置故障转移，每个 Master 有 Slave
内存上限受单机内存限制多机内存总和
运维复杂度较低较高
客户端支持普通客户端即可需要支持 Cluster 协议的客户端
适用场景数据量不大，读写分离数据量大，需要水平扩展
批量操作支持 MGET 等批量命令跨槽操作需 hash tag

四、生产环境注意事项

4.1 持久化

  • RDB：定时快照，恢复快，但可能丢失最近数据
  • AOF：每秒 fsync，数据更安全，但文件更大
  • 推荐：同时开启 RDB + AOF，AOF 为主，RDB 用于快速恢复

4.2 内存管理

淘汰策略选择：

策略说明适用场景
noeviction不淘汰，内存满拒绝写入数据不能丢失
allkeys-lru所有 key 中淘汰最久未使用的缓存场景
volatile-lru设置了过期时间的 key 中淘汰部分数据可丢失
allkeys-random随机淘汰无明显热点
volatile-ttl淘汰 TTL 最短的业务有明确过期需求

五、常见问题

Q1：主从复制延迟怎么处理？

  • 使用 REPLCONF ACK 确认从节点同步进度
  • 对一致性要求高的读操作走 Master
  • 配置 min-replicas-to-write 1，至少 1 个从节点同步成功才允许写入

总结

场景推荐方案
数据量 < 单机内存，需要读写分离一主两从 + Sentinel
数据量 > 单机内存，需要水平扩展Redis Cluster
对一致性要求极高一主两从 + Sentinel（读走主）
高并发写入Redis Cluster（分摊写入压力）

选择方案的核心依据是数据量和扩展需求。数据量不大时，Sentinel 方案更简单可靠；数据量大或需要动态扩展时，Cluster 是唯一选择。

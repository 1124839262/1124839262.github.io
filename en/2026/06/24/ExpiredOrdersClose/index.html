Overview

Expired order closing is a common business requirement in e-commerce systems. When a user places an order but doesn’t pay within a specified time, the order needs to be automatically closed and inventory released.

Solution 1: Scheduled Task (Essential Fallback)

Use a scheduled task framework (e.g., XXL-Job) to periodically scan expired unpaid orders.

Implementation

Pros and Cons

ProsCons
Simple and reliableMinute-level delay, not real-time
Supports distributed deploymentScans large amounts of data each time
Natural fallback mechanismPuts pressure on database

Best Practices

Standard Scenario (Delay ≤ 3 days)

RocketMQ Delayed Message + XXL-Job Scheduled Task Fallback

  • RocketMQ for real-time processing, second-level precision
  • XXL-Job scans hourly, only as fallback for missed orders
  • 4.x: Use directly for delays ≤ 2h, circular delivery for > 2h
  • 5.x: Specify arbitrary delay duration directly

Long Delay Scenario (Delay > 3 days)

Redisson Delay Queue + XXL-Job Scheduled Task Fallback

  • For scenarios like 7-day coupon expiration, RocketMQ’s default 3-day retention may lose messages
  • Redisson supports arbitrary delay durations, independent of message retention time
  • XXL-Job scheduled task as final fallback
  • If increasing RocketMQ retention time (fileReservedTime), evaluate disk space

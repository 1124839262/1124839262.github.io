Overview

In production environments, Redis typically does not run as a single node. Instead, Master-Slave Replication or Cluster mode is used to ensure high availability and data safety. This article covers two mainstream deployment approaches:

  • One Master, Two Slaves + Sentinel: Suitable for read-write splitting and high availability
  • Redis Cluster: Suitable for large datasets requiring horizontal scaling

1. One Master, Two Slaves + Sentinel Deployment

1.1 Architecture

  • Master: Handles both read and write operations
  • Slave: Read-only, synchronizes data from Master
  • Sentinel: Monitors master-slave status and performs automatic failover

2. Redis Cluster Deployment

2.1 Architecture

Redis Cluster divides data into 16384 slots, distributed across Master nodes. Each key is assigned to a slot via CRC16(key) % 16384.

2.2 Minimum Cluster Requirements

  • At least 6 nodes: 3 Masters + 3 Slaves
  • Each Master requires at least 1 Slave for high availability

3. Comparison of the Two Approaches

FeatureMaster-Slave + SentinelRedis Cluster
Data ShardingNot supported; all data on MasterSupported; data distributed across Masters
Horizontal ScalingNot supported; vertical scaling onlySupported; add nodes and reshard
High AvailabilitySentinel automatic failoverBuilt-in failover; each Master has a Slave
Memory LimitLimited by single machine memorySum of all machine memories
Operational ComplexityLowerHigher
Client SupportStandard client sufficientRequires Cluster-protocol-aware client
Best ForModerate data, read-write splittingLarge data, horizontal scaling needed
Batch OperationsSupports MGET and similar commandsCross-slot operations require hash tags

4. Production Best Practices

4.1 Persistence

  • RDB: Periodic snapshots; fast recovery but potential data loss
  • AOF: fsync every second; safer data but larger files
  • Recommendation: Enable both RDB + AOF; use AOF as primary, RDB for fast recovery

4.2 Memory Management

Eviction policy selection:

PolicyDescriptionUse Case
noevictionNo eviction; reject writes when memory is fullData cannot be lost
allkeys-lruEvict least recently used keys among all keysCaching scenarios
volatile-lruEvict LRU among keys with TTL setSome data can be lost
allkeys-randomEvict random keysNo obvious hot keys
volatile-ttlEvict keys with shortest TTLBusiness has explicit expiration needs

5. FAQ

Q1: How to handle master-slave replication lag?

  • Use REPLCONF ACK to check slave sync progress
  • Route consistency-critical reads to the Master
  • Configure min-replicas-to-write 1 to require at least 1 slave to be in sync before accepting writes

Summary

ScenarioRecommended Approach
Data fits in single machine memory, need read-write splittingMaster-Slave + Sentinel
Data exceeds single machine memory, need horizontal scalingRedis Cluster
High consistency requirementsMaster-Slave + Sentinel (read from Master)
High concurrent writesRedis Cluster (distributes write pressure)

The core criteria for choosing an approach are data volume and scaling requirements. For moderate data sizes, the Sentinel approach is simpler and more reliable; for large data or dynamic scaling needs, Cluster is the only option.

Preface

Today I attended a Java backend developer interview. The interviewer asked in-depth questions across five areas: Java fundamentals, MySQL, JVM tuning, Redis caching, and message queues. This article documents all the questions with detailed reference answers — both as a personal review and for future study reference.

1. Java Fundamentals

1. What implementation classes does Map have?

Common implementation classes of Java’s Map interface:

ImplementationThread-SafeOrderingAllows null keyAllows null valueUnderlying Structure
HashMapNoUnorderedYesYesArray + LinkedList + RBTree
LinkedHashMapNoInsertion/Access orderYesYesHashMap + DoublyLinkedList
TreeMapNoSorted by key (natural)NoYesRed-Black Tree
HashtableYes (synchronized)UnorderedNoNoArray + LinkedList
ConcurrentHashMapYes (CAS+synchronized)UnorderedNoNoArray + LinkedList + RBTree

There are also less commonly used implementations like IdentityHashMap, WeakHashMap, and EnumMap.

2. What is the underlying data structure of HashMap?

In JDK 1.8, HashMap’s underlying data structure is Array + LinkedList + Red-Black Tree:

  • Array: Node<K,V>[] table, default initial capacity 16, hash function determines bucket position
  • LinkedList: On hash collision, elements in the same bucket are chained (tail insertion; JDK 1.7 used head insertion)
  • Red-Black Tree: When a bucket’s linked list length exceeds 8 AND the array capacity ≥ 64, the linked list converts to a red-black tree, reducing lookup from O(n) to O(log n); when tree nodes ≤ 6, it degrades back to a linked list

Key parameters:

  • Default initial capacity: 16 (DEFAULT_INITIAL_CAPACITY)
  • Load factor: 0.75 (DEFAULT_LOAD_FACTOR)
  • Treeification threshold: 8 (TREEIFY_THRESHOLD)
  • Untreeification threshold: 6 (UNTREEIFY_THRESHOLD)
  • Minimum treeification capacity: 64 (MIN_TREEIFY_CAPACITY)

3. Is HashMap thread-safe?

HashMap is NOT thread-safe. Using HashMap in a multi-threaded environment can cause various issues:

  • JDK 1.7: Concurrent resizing with head insertion can cause linked list cycles, leading to infinite loops in get() operations
  • JDK 1.8: Tail insertion resolves the cycle issue, but data overwriting and inaccurate size still occur

Thread-safe alternatives:

  • ConcurrentHashMap (recommended, better performance)
  • Collections.synchronizedMap(new HashMap<>()) (global lock, poor performance)
  • Hashtable (deprecated, not recommended)

4. What problems occur with concurrent multi-threaded operations on HashMap?

ProblemDescription
Data overwritingTwo threads put to the same bucket simultaneously; the later write overwrites the earlier one, causing data loss
Inaccurate sizeMultiple threads modifying size concurrently; ++size is not atomic, count may be wrong
Data loss during resizingMultiple threads triggering resize simultaneously; some nodes may be lost during migration
Infinite loop (JDK 1.7)Concurrent resizing with head insertion causes linked list cycle; get() enters infinite loop, CPU spikes to 100%
ConcurrentModificationExceptionAnother thread modifies structure during iteration, triggering fail-fast mechanism

5. How does ConcurrentHashMap ensure thread safety?

JDK 1.7 — Segment Locking

  • Data divided into 16 Segments, each Segment is a small Hashtable
  • Operations on different Segments don’t affect each other; default concurrency level is 16
  • Lock granularity: Segment level

JDK 1.8 — CAS + synchronized

  • Segment locks removed; operations directly on the Node array
  • put flow:
    1. Compute hash to locate bucket
    2. If bucket is empty, use CAS to attempt write (lock-free)
    3. If bucket is not empty, lock the head node with synchronized then operate
    4. If resizing is in progress, help with migration (helpTransfer)
  • Lock granularity: bucket head node level, finer than segment lock, concurrency equals array length

Other thread-safety mechanisms:

  • sizeCtl variable controls resizing via CAS
  • CounterCell array for size counting, avoiding contention
  • volatile on Node’s val and next, ensuring visibility

7. What are the parameters of a thread pool?

7 core parameters of ThreadPoolExecutor:

ParameterDescription
corePoolSizeCore thread count; not reclaimed even when idle
maximumPoolSizeMaximum thread count; upper limit of core + non-core
keepAliveTimeIdle survival time for non-core threads
unitTime unit for survival time
workQueueTask waiting queue, e.g., LinkedBlockingQueue, SynchronousQueue
threadFactoryThread factory for customizing thread names, daemon status, etc.
handlerRejection policy; how to handle tasks when queue is full and threads are maxed out

Four rejection policies:

  • AbortPolicy (default): throws RejectedExecutionException
  • CallerRunsPolicy: the submitting thread executes the task itself
  • DiscardPolicy: silently discards the task without exception
  • DiscardOldestPolicy: discards the oldest task in the queue and resubmits the current task

8. How are thread pool parameters typically configured in production?

Production configuration principles:

  1. Core pool size (corePoolSize):

    • CPU-intensive tasks: CPU cores + 1
    • IO-intensive tasks: CPU cores * 2 or CPU cores / (1 - blocking coefficient)
    • Actual values determined through load testing

  2. Maximum pool size (maximumPoolSize): Based on system capacity and task type, typically ~2x core pool size

  3. Queue capacity: Use bounded queue (e.g., LinkedBlockingQueue(500)) to avoid OOM from unbounded queue

  4. Custom thread factory: Set meaningful thread names for easier troubleshooting

  1. Rejection policy: Typically CallerRunsPolicy — doesn’t discard tasks, doesn’t throw exceptions, and provides backpressure

  2. Dynamic adjustment: Use config center (Nacos/Apollo) for dynamic parameter modification, with monitoring (active threads, queue size, rejection count) for real-time tuning

2. MySQL Database

1. What database do you use daily? Have you used MySQL?

I primarily use MySQL daily, and have also worked with SQL Server and PostgreSQL. MySQL is the most popular open-source relational database and is widely used in internet companies.

2. What data structure does MySQL index use?

MySQL InnoDB engine indexes use B+ Tree as the underlying data structure.

B+ Tree characteristics:

  • Non-leaf nodes store only index keys, not data, allowing more keys per node and making the tree shorter and wider
  • All data is stored in leaf nodes
  • Leaf nodes are connected via a doubly-linked list, supporting efficient range queries and sequential scans
  • Tree height is typically 3~4 levels; tens of millions of records require only 3~4 IOs

3. What are the advantages of using B+ Tree for MySQL indexes?

Compared to B Tree, Hash index, and binary tree:

ComparisonB+ TreeB TreeHash IndexBinary Tree
Range queryEfficient (sequential scan via leaf linked list)Requires in-order traversalNot supportedRequires in-order traversal
Equality queryO(log n)O(log n)O(1)O(log n)
SortingSupported (ordered)Partially supportedNot supportedSupported
Disk IOLow (3~4 times)HigherLowHigh (tall tree)
Node capacityLarge (non-leaf stores only keys)Small (stores keys + data)—Small

Core advantages summary:

  1. Fewer IOs: Non-leaf nodes don’t store data, high fan-out, short tree, queries need only 3~4 disk IOs
  2. Efficient range queries: Leaf nodes linked together; sequential scan after finding the start point
  3. Stable queries: All data in leaf nodes; every query path has the same length

4. What database query optimizations have you done?

Index level:

  • Follow the leftmost prefix rule to avoid index invalidation
  • Use covering indexes to reduce table lookups
  • Avoid using functions or implicit type conversions on indexed columns
  • Place high-cardinality columns first in composite indexes
  • Use prefix indexes for long strings

SQL level:

  • Avoid SELECT *; query only needed columns
  • Small table drives large table; optimize JOIN order
  • Use EXISTS instead of IN for large datasets
  • Avoid operations on indexed columns in WHERE clause
  • Deep pagination optimization: use WHERE id > lastId instead of LIMIT offset, size

Architecture level:

  • Read-write separation: master for writes, slaves for reads
  • Sharding (ShardingSphere): split by business or data volume
  • Introduce Redis caching for hot data

Diagnostic tools:

  • EXPLAIN to analyze execution plans, focus on type, key, rows, Extra
  • Slow query log to identify slow SQL
  • SHOW PROFILE to analyze SQL execution time by phase

5. What are the four properties of transactions?

The four properties of transactions are known as ACID:

PropertyMeaningImplementation Mechanism
AtomicityAll operations in a transaction either all succeed or all rollbackundo log
ConsistencyData transitions from one consistent state to another after transaction executionGuaranteed by Atomicity + Isolation + Durability
IsolationConcurrent transactions don’t interfere with each otherLock mechanism + MVCC
DurabilityAfter commit, modifications are permanently savedredo log

MySQL InnoDB default isolation level is REPEATABLE READ, implemented via MVCC for snapshot reads and gap locks to prevent phantom reads.

3. Production system suddenly runs slowly, APIs keep loading — what are possible causes and how to troubleshoot?

Possible causes:

CategoryCause
GC issuesFrequent Full GC / long GC pauses, STW blocking threads
Memory issuesMemory leak leading to pre-OOM state, triggering frequent Full GC
Thread issuesDeadlock, thread pool full, many threads BLOCKED/WAITING
Database issuesSlow SQL, connection pool full, lock waits
External dependenciesDownstream service timeout, network issues
Resource issuesCPU spike, high disk IO, network bandwidth saturated

Troubleshooting steps:

  1. Check system resources: top / htop for CPU usage, free for memory, iostat for disk IO

  2. Find Java process: jps to get PID

  3. Check GC status:

    Focus on FGC count and FGCT time; frequent Full GC with long duration indicates a problem

  4. Check thread stacks:

    Search for BLOCKED, WAITING, deadlock keywords

  5. Check memory usage:

  6. Export heap dump for analysis:

    Use MAT or VisualVM to analyze, find memory leaks and large objects

  7. Analyze GC logs: Use GCViewer or GCEasy to analyze GC logs, confirm pause times and frequency

  8. APM tools: SkyWalking / Arthas for online diagnosis, trace command to track method execution time

4. Redis Caching

1. Have you used Redis?

Yes, Redis is one of the most frequently used middleware in our projects, primarily for caching, distributed locks, rate limiting counters, leaderboards, and more.

2. What data structures does Redis have?

Redis has 5 basic data structures and 3 extended data structures:

TypeUnderlying EncodingCommon Uses
Stringint / embstr / rawCaching, counters, distributed locks
Listquicklist (ziplist + linkedlist)Message queue, article list
Hashziplist / hashtableObject storage (user info, etc.)
Setintset / hashtableTags, mutual friends, deduplication
ZSetziplist / skiplist+hashtableLeaderboards, delay queues
HyperLogLog—Cardinality estimation (UV counting)
Bitmap—Check-in, Bloom filter
GeoZSetGeolocation, nearby people

3. What is the underlying data structure of ZSet?

ZSet (sorted set) has two underlying encodings:

  1. ziplist (compressed list): Used when element count ≤ 128 and each element length ≤ 64 bytes; memory-efficient, suitable for small datasets

  2. skiplist + hashtable: Used when ziplist conditions are not met

    • skiplist: Supports range queries, O(log n) lookup, ordered by score
    • hashtable: Stores member → score mapping, supports O(1) lookup by member

4. What persistence methods does Redis have?

MethodPrincipleProsCons
RDBPeriodically snapshot memory data to disk (dump.rdb)Compact file, fast recovery, low performance impactMay lose data between snapshots
AOFAppend write commands to log file (appendonly.aof)Safer data, at most 1 second data lossLarge file, slow recovery, performance impact
Mixed persistenceRDB+AOF hybrid; rewrite uses RDB format for first half, AOF for second halfBalanced recovery speed and data safetyRequires Redis 4.0+

Production recommendation: Enable AOF + mixed persistence, keep RDB as backup.

5. Redis AOF command log file keeps growing — how to optimize?

The core reason AOF files grow is that commands are only appended, never removed. Even if a key is deleted or overwritten, old commands remain.

Optimization: AOF rewrite

AOF rewrite reads the current database state and regenerates the AOF file with minimal commands:

Trigger methods:

  • Manual trigger: BGREWRITEAOF command
  • Auto trigger: Configure auto-aof-rewrite-percentage and auto-aof-rewrite-min-size

With mixed persistence enabled in Redis 4.0+, rewrite uses RDB format for the first half, significantly reducing file size.

6. How to ensure data consistency between Redis cache and database?

Cache-database consistency is a classic challenge. Common strategies:

Cache Aside Pattern (most commonly used):

  • Read: Read cache first; if hit, return; if miss, read database, write to cache
  • Write: Update database first, then delete cache

Why delete cache instead of updating cache?

  • Avoids inconsistency from concurrent writes
  • Deletion is simpler; lazy load on next read

Why update database first, then delete cache?

  • Delete cache first then update database: concurrent reads may cache stale data
  • Update database first then delete cache: brief inconsistency possible in extreme concurrency, but probability is low and time window is minimal

Strong consistency solutions (choose as needed):

  • Delayed double delete: delete cache → update database → delay N ms → delete cache again
  • Listen to binlog: Canal monitors MySQL binlog, asynchronously deletes/updates cache
  • Distributed transaction: Seata etc. for strong consistency (high performance cost)

5. Message Queues

1. Have you used message middleware?

Yes, I’ve used RocketMQ and RabbitMQ in projects:

  • RocketMQ: For async order processing, delayed messages (auto-cancel unpaid orders), transaction messages, etc.
  • RabbitMQ: For log collection, async notifications, and other lightweight scenarios

2. Has RocketMQ ever experienced message loss?

RocketMQ can lose messages at three stages:

StageCause of LossSolution
① Producer sendNetwork error, send failed without retrySync send + retry mechanism (retryTimesWhenSendFailed)
② Broker storageAsync flush; data in memory lost on crashSync flush (FLUSH_SYNC) + sync replication (SYNC_MASTER)
③ Consumer consumeAuto-ACK on consumption failureManual ACK: submit only after successful business processing

  • Enable message trace to track the full message lifecycle
  • Dead letter queue (DLQ) for messages that fail after multiple retries
  • Idempotent checks on the business side to prevent duplicate consumption

Why Rate Limiting is Needed

In high-concurrency scenarios, system resources (CPU, memory, connections, database connection pools) are limited. Without rate limiting protection, burst traffic can cause:

  • Service cascading failure: Backend services get overwhelmed, affecting all users
  • Database overload: Massive requests hitting the DB simultaneously, causing slow query accumulation
  • Bandwidth waste: Malicious crawlers or abnormal traffic consuming bandwidth resources

The core goal of rate limiting: Process as many legitimate requests as possible while ensuring service quality.

I. Fixed Window Counter

Algorithm Principle

Divide time into fixed windows (e.g., 1 second, 1 minute), with independent counting within each window.

OpenResty Implementation

Pros and Cons

ProsCons
Simple implementation, high performanceBoundary issue: Up to 2x burst at window edges
Constant memory usageCannot handle burst traffic smoothly

Boundary issue example: Assuming a limit of 100 requests/minute, 100 requests at 00:59, window resets at 01:00 allowing another 100, effectively processing 200 requests in 1 second.

II. Sliding Window Counter

Algorithm Principle

Instead of splitting by fixed boundaries, record each request’s timestamp and count requests within the last N milliseconds/seconds.

Pros and Cons

ProsCons
Solves the boundary issueMemory grows with request volume

Supports dynamic window adjustmentJSON encoding/decoding has some CPU overhead

III. Leaky Bucket Algorithm

Algorithm Principle

Imagine a bucket with a hole at the bottom:

  • Requests (water) flow into the bucket from the top
  • Water leaks out at a constant rate from the bottom (processing requests)
  • If the bucket is full, new requests are rejected

Core characteristic: Output rate is constant, enabling forced traffic smoothing.

OpenResty Implementation

Use Cases

ScenarioDescription
Protect downstream servicesForce limit rate to backend, prevent overload
Traffic shapingSmooth burst traffic into constant output
API gateway rate limitingEnsure backend service stability

Pros and Cons

ProsCons
Constant output rate, ideal for backend protectionCannot handle burst traffic (even with spare bucket capacity)
Relatively simple implementationRequires precise leak rate calculation
Fixed memory usageParameter tuning requires experience

IV. Token Bucket Algorithm

Algorithm Principle

The system adds tokens to the bucket at a constant rate:

  • Each request consumes 1 token
  • Has tokens → Allow through
  • No tokens → Reject or wait
  • When bucket is full, excess tokens overflow and are discarded

Core characteristic: Allows a certain degree of burst traffic (tokens accumulated in the bucket).

OpenResty Implementation

Use Cases

ScenarioDescription
APIs allowing burst trafficE.g., users rapidly clicking buttons
CDN/edge nodesBalance user experience and system protection
Microservice callsAllow short bursts but control average rate

Pros and Cons

ProsCons
Allows reasonable burstsNeeds time to recover after burst
Intuitive and easy-to-understand parametersLong idle periods may cause large initial burst
Widely adopted (Google Guava, Envoy, etc.)Requires proper rate and burst configuration

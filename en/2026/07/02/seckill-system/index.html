Overview

A seckill (flash sale) system is a typical high-concurrency scenario. The core challenge is handling massive traffic spikes while ensuring:

  • No overselling
  • Fairness
  • System stability

I. Overall Architecture

II. Traffic Layered Defense

Layer 1: CDN / Static Resource Offloading

  • Static pages (product details, event pages) are deployed on CDN
  • Users browse without hitting the backend at all

Layer 2: Nginx Rate Limiting

Layer 3: Gateway Filtering

  • User authentication and authorization
  • Interface hiding: seckill URL dynamically generated with time-limited tokens
  • Blacklist checking

Layer 4: Application Layer

  • Pre-deduct inventory in Redis
  • MQ async order placement
  • Distributed lock for duplicate prevention

III. Inventory Pre-deduction

Option 1: Redis DECR (Simple & Efficient)

Option 2: Redis + Lua Script (More Rigorous)

Combines the check and deduction into a single atomic operation, avoiding concurrency window issues:

  • Returns 1: Deduction successful
  • Returns -1: Insufficient stock
  • Returns -2: Duplicate order

IV. Async Order Placement

MQ Peak Shaving

  • Seckill service only handles pre-deduction + sends message, returns quickly
  • Order service consumes messages, creates orders, deducts database stock
  • When downstream processing capacity is insufficient, MQ naturally buffers, protecting the database

User-level Rate LimitingSame user can only purchase the same product once
Hidden APISeckill URL dynamically generated with time-limited token
BlacklistRisk control system identifies abnormal users
Hidden StockFrontend doesn’t show real-time stock, only shows available/sold out

VII. Stock Rollback

Stock needs to be replenished in the following scenarios:

  1. Order timeout without payment -> Close order, replenish stock
  2. Order creation failed (DB exception) -> Replenish Redis stock
  3. User cancels order -> Replenish stock

Note: After replenishing stock, check if the item needs to be re-opened for purchase (may have been marked as sold out).

VIII. Data Consistency Reconciliation

MechanismDescription
Real-time reconciliationRedis stock + DB ordered count = Total stock
Scheduled taskScan every minute, alert and fix if inconsistent
After event endsClear Redis stock, use DB as source of truth for final calibration

IX. High Availability

  • Multi-instance deployment with Nginx load balancing
  • Redis cluster (Sentinel or Cluster), avoid single point of failure
  • MQ cluster, ensure no message loss
  • Degradation switch: After stock is sold out, return directly at gateway level
  • Circuit breaker: When downstream services fail, fail fast to avoid cascading failures

Summary: Core Principles of Seckill Systems

  1. Intercept requests as early as possible – Return at CDN if possible, otherwise Nginx, otherwise gateway
  2. Cache for reads, MQ for writes – Reads go to Redis, writes go through MQ for async processing
  3. Atomic stock deduction – Redis Lua script or DECR atomic operation
  4. Database as final safeguard – stock > 0 condition prevents overselling
  5. Async eventual consistency – Accept brief inconsistency, ensure eventual consistency through reconciliation
  6. Degradable and circuit-breakable – Better to undersell than oversell in extreme cases

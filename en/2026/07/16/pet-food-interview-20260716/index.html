Yesterday afternoon I interviewed at a pet food company. The interviewer covered a very comprehensive range of topics — from business concepts to database tuning, from microservice architecture to RocketMQ deep dives, basically going through the entire Java backend knowledge system. I answered most of the questions, but some got shaky when pressed deeper. Let me organize everything while it’s still fresh, with reference answers for future review.

1. Manufacturing Business Concepts

What are the differences between MES, OMS, and WMS?

Core system concepts in manufacturing/warehousing. The interviewer wants to confirm your business understanding.

SystemFull NameCore ResponsibilityKey Functions
MESManufacturing Execution SystemShop-floor production process controlProduction scheduling, work order management, process routing, quality traceability, equipment monitoring, reporting
OMSOrder Management SystemFull order lifecycle managementOrder entry, review, split/merge, allocation, fulfillment tracking, returns
WMSWarehouse Management SystemInternal warehouse operationsInbound, outbound, inventory counting, transfer, location management, batch management, picking strategy

One-liner: OMS manages “what to sell” → MES manages “how to make it” → WMS manages “where to store it”. Together they form the complete chain from order to production to warehousing in a manufacturing enterprise.

2. LIMS Local AI Project Deep Dive

The interviewer was very interested in the AI project, asking 4 consecutive questions — from deployment approach to data sources to architecture design, digging quite deep.

1. Is the AI calling third-party APIs or locally deployed?

Reference answer: Locally deployed. We use Ollama as the model runtime framework, deploying the Qwen2 large model on the server. All inference is done locally without relying on external APIs. Two reasons for local deployment: first, data security — lab data cannot be sent externally; second, network isolation — the production environment cannot access the internet.

2. Is the Agent locally deployed? Did you do the model fine-tuning?

Reference answer: The Agent is also locally deployed. For model fine-tuning, we didn’t do fine-tuning of the base model because we didn’t have enough compute resources or training data. Instead, we optimized using the RAG (Retrieval-Augmented Generation) approach:

  • Knowledge base document chunking strategy optimization (chunking by paragraph/chapter, controlling chunk size to 500-1000 tokens)
  • Retrieval strategy optimization (hybrid retrieval: vector search + keyword search, fused with RRF algorithm)
  • Prompt template optimization (adding role definition, response constraints, source citation requirements)

This approach doesn’t require model training, but the effect was significant — answer accuracy improved from 60% in the initial version to over 85%.

3. What are the knowledge base data sources? How do you handle the vector database when files are updated?

Reference answer: The knowledge base data comes from three main sources:

  • SOP documents: Standard Operating Procedures, in PDF and Word formats
  • Testing standards: National and industry standards, in PDF format
  • Equipment manuals: Instrument operation manuals, in PDF and Word formats

The update process when files change:

  1. When a document is updated, the system detects the change via file MD5
  2. Delete all vector chunks associated with the old document (linked by document ID)
  3. Re-parse the new document, chunk it, and generate new vector embeddings
  4. Write the new vectors to Milvus, linking to the new document ID and version number
  5. Log the update with support for rollback to historical versions

4. How do you handle multi-type device data (push/Excel/Word)?

Reference answer: We designed a unified data ingestion layer with different adaptation strategies for different data sources:

Data Source TypeIngestion MethodProcessing Logic
Device push (HTTP/MQTT)Unified data receiving interface supporting REST and MQTT protocolsData validation → format conversion → write to database
Excel filesScheduled directory scan or manual upload triggerEasyExcel streaming read → data validation → batch write
Word filesManual upload triggerApache POI parsing → extract structured data → write to database

The core design uses the Strategy pattern: define a unified DataProcessor interface, with each data source implementing its own processor. A factory class retrieves the appropriate processor based on data source type. Adding a new data source type only requires adding a new implementation class — no changes to existing code.

5. Your most recent role was project manager — do you still write code?

The interviewer is concerned you may have been away from coding too long. Emphasize that your coding ability hasn’t degraded.

Reference answer: Although my title was project manager, the team was small, so I’ve always been doing both technical and management work. I spend over 50% of my time writing code. Core modules and technical challenges are handled by me personally. For example, the AI integration module in LIMS was independently designed and implemented by me from architecture to coding.

3. Database Questions (SQL Server/MySQL)

Database questions were very detailed — from tuning techniques to massive data handling to lock mechanisms, basically covering all MySQL core knowledge points.

1. What practical database tuning and optimization techniques have you used?

This is a broad question. Organize your answer across multiple dimensions to appear structured.

Reference answer:

SQL level:

  • Slow query log analysis to identify problem queries
  • EXPLAIN to analyze execution plans — focus on type, key, rows, Extra fields
  • Avoid SELECT * — only query needed columns
  • Use indexes properly, following the leftmost prefix rule
  • Avoid function operations or implicit type conversions on indexed columns
  • For large table JOINs, ensure join fields are indexed; drive with the smaller table
  • Deep pagination optimization: use WHERE id > last_id LIMIT n instead of LIMIT offset, n

Index level:

  • Index high-frequency query fields; don’t index low-selectivity fields (like gender) alone
  • For composite indexes, pay attention to column order — higher cardinality first
  • Covering indexes to reduce table lookups
  • Regularly clean up redundant indexes

Table structure level:

  • Field type selection: DECIMAL for money, TINYINT for status, DATETIME for timestamps
  • Appropriate denormalization to reduce JOINs
  • Split large fields into extension tables

Configuration level:

  • innodb_buffer_pool_size set to 60%-80% of physical memory
  • innodb_flush_log_at_trx_commit adjusted based on business requirements
  • sync_binlog balanced between safety and performance
  • Connection pool parameter tuning

2. Have you handled tens of millions or hundreds of millions of device data records? How?

Reference answer: Yes. Device data collection volume was very large, with single tables quickly reaching tens of millions. Our approach:

  • Hot/cold separation: Recent 3 months of hot data in the main database; historical data archived to history tables or data warehouse
  • Table sharding: Monthly tables by time dimension, e.g., device_data_202607; queries route to the appropriate table based on time
  • Read/write separation: Writes go to the primary; report queries go to replicas
  • Data compression: Historical tables use columnar storage (ClickHouse) with up to 10:1 compression ratio
  • Sampling storage: Second-level data retained for 7 days, minute-level aggregated data for 3 months, hour-level aggregated data permanently

3. What solutions exist for hot data exceeding 20 million rows?

Reference answer:

SolutionApplicable ScenarioDescription
Index optimizationSlow queries but manageable data volumeAdd appropriate indexes, covering indexes, index condition pushdown
Partitioned tablesData has a natural partition key (like time)RANGE partitioning; queries automatically prune partitions
ShardingClear single-table performance bottleneckShardingSphere middleware; shard by tenant/time
Read/write separationRead-heavy, write-lightPrimary for writes, replicas for reads
CachingHot data frequently readRedis cache for query results, reducing DB pressure
Cold data archivalLarge proportion of historical dataScheduled tasks to migrate cold data, keeping hot tables lean

In practice, solutions are typically combined. Our approach was: partitioned tables + read/write separation + hot/cold separation + Redis caching.

4. Have you done database/table sharding? What solutions do you know?

Reference answer: I’ve done table sharding but not database sharding. We used ShardingSphere-JDBC, a client-side sharding middleware that doesn’t require deploying a separate proxy service.

Sharding strategy:

  • Table sharding: Monthly tables, sharding key is create_time, using RANGE sharding algorithm
  • Sharding rules: Defined in YAML configuration file mapping logical tables to physical tables
  • Cross-table queries: ShardingSphere automatically routes and merges results, transparent to business code

If data volume grows to require database sharding, the approach would consider:

  • Shard by business domain: user DB, order DB, device DB each independent
  • Sharding key selection: Prioritize fields that distribute data evenly, avoiding data skew
  • Distributed ID: Snowflake algorithm for globally unique IDs
  • Cross-database JOINs: Assemble at the code level, or use wide tables with redundant fields to avoid cross-database JOINs

5. Do you have experience handling MySQL connection issues?

Reference answer: Yes. A project encountered Too many connections errors. The troubleshooting process:

  1. SHOW PROCESSLIST to check current connection status — found many Sleep idle connections
  2. Checked application-side connection pool config — found max connections set too high (500) while MySQL’s max_connections was only 200
  3. Connection pool’s maxIdle was not configured, so idle connections were never released

Solution:

  • Adjusted connection pool parameters: maxActive=50, maxIdle=20, minIdle=10, maxWait=3000
  • MySQL side: max_connections=500, wait_timeout=600 (10-minute idle disconnect)
  • Enabled connection pool monitoring to regularly check active connections and waiting threads

Key parameter reference:

ParameterDescriptionSuggested Value
max_connectionsMySQL max connectionsCalculate based on memory; ~1MB per connection
wait_timeoutNon-interactive connection idle timeout600-1800 seconds
interactive_timeoutInteractive connection idle timeoutSame as wait_timeout
max_connect_errorsMax connection errorsPrevents IP from being blocked

4. Microservices & Spring Cloud Architecture

This section was quite varied — from tech stack introduction to component comparison to startup process, spanning a wide range.

1. Describe your familiar tech stack

Reference answer: Core tech stack is Spring Boot + Spring Cloud Alibaba microservices ecosystem:

  • Frameworks: Spring Boot 2.x / 3.x, Spring Cloud 2021
  • Registry/Config Center: Nacos
  • Gateway: Spring Cloud Gateway
  • RPC: OpenFeign (synchronous), Dubbo (familiar, not deeply used)
  • Rate Limiting/Circuit Breaking: Sentinel
  • Message Queue: RocketMQ
  • Databases: MySQL, SQL Server
  • Cache: Redis
  • Search Engine: Elasticsearch
  • Containerization: Docker
  • Monitoring: Prometheus + Grafana

2. What’s the essential difference between Spring Cloud Feign and Dubbo?

Reference answer:

ComparisonOpenFeignDubbo
ProtocolHTTP/HTTPSCustom Dubbo protocol (default TCP long connection)
CommunicationRESTful, HTTP-basedRPC, TCP-based
SerializationJSON (default)Hessian2, Protobuf, etc. (binary)
PerformanceLower (HTTP overhead + JSON serialization)Higher (long connection + binary serialization)
EcosystemSpring Cloud native, easy integrationAlibaba ecosystem, feature-rich
Use caseExternal APIs, cross-language callsInternal high-performance service calls

Essential difference: Feign is an HTTP-based declarative REST client operating at the application layer — good interoperability but average performance. Dubbo is a custom-protocol RPC framework operating at the transport layer — high performance but tied to the Java ecosystem. If all services are Java and performance-sensitive, choose Dubbo; if you need cross-language or external access, choose Feign.

3. What component do you use for rate limiting?

Reference answer: We use Sentinel. Alibaba’s open-source traffic control component supporting multiple rate limiting strategies:

  • QPS limiting: Control requests per second
  • Thread count limiting: Control concurrent threads
  • Circuit breaking: Triggered by slow call ratio / error ratio / error count
  • Hotspot parameter limiting: Rate limit by parameter value, e.g., per user ID

Configuration supports both hard-coded and Nacos dynamic configuration. We use the Nacos approach so rate limiting rules can be changed without restarting services.

5. What’s the difference between Nacos and Eureka?

Reference answer:

Core difference: Nacos supports both AP and CP modes — ephemeral instances use AP (Distro protocol), persistent instances use CP (Raft protocol). Eureka only supports AP mode. Additionally, Nacos integrates config center capabilities, eliminating the need for Spring Cloud Config.

6. Can Elasticsearch be used as a config center?

Reference answer: Technically yes, but not recommended. ES’s core strength is full-text search and data analytics, not configuration management. Using it as a config center has these issues:

  • No real-time config change push mechanism; requires polling
  • No config version management or rollback
  • No permission control or canary deployment
  • Config data is small; using ES is overkill and wastes resources

Config centers should use purpose-built components: Nacos or Apollo.

7. Do you know Spring Cloud Bus?

Reference answer: Yes. Spring Cloud Bus propagates state changes across microservice nodes, built on top of message brokers (RabbitMQ/Kafka).

A typical use case is working with Spring Cloud Config for dynamic config refresh: when config changes in the Git repo, Config Server sends a refresh event through Bus to all services, which then pull the latest config without restarting.

However, Nacos has built-in config push capabilities, so Bus is largely unnecessary now. Bus is more of a Spring Cloud Netflix-era artifact with limited use cases in the Spring Cloud Alibaba ecosystem.

8. What are the differences and advantages of Spring Boot vs traditional SSM?

Reference answer:

Dependency managementManual version management; conflicts commonStarter auto-manages versions
Embedded containerRequires external Tomcat deploymentEmbedded Tomcat/Jetty; run with java -jar

Core advantages summary: Spring Boot solves the “configuration hell” of traditional SSM through three core features: auto-configuration (@EnableAutoConfiguration), starter dependencies, and embedded containers — letting developers focus on business logic.

9. Do you understand the Spring Boot startup process?

Reference answer: The Spring Boot startup process is roughly:

  1. Execute main method: Call SpringApplication.run()
  2. Create SpringApplication instance:
    • Infer application type (Servlet/Reactive/None)
    • Load ApplicationContextInitializer and ApplicationListener
    • Infer the main configuration class
  3. run method execution:
    • Create BootstrapContext
    • Get and start SpringApplicationRunListener
    • Prepare Environment: read config files, environment variables
    • Create ApplicationContext (different container implementations based on application type)
    • Prepare context: execute ApplicationContextInitializer, register BeanDefinitions
    • Refresh context (core step):
      • BeanDefinition scanning and registration (@ComponentScan, @Import)
      • BeanFactoryPostProcessor execution (handles @ConfigurationProperties binding, etc.)
      • Register BeanPostProcessor
      • Instantiate all singleton beans (resolving circular dependencies)
    • Execute afterRefresh: call CommandLineRunner and ApplicationRunner
    • Publish ApplicationReadyEvent
  4. Application startup complete

The core mechanism is auto-configuration: through @EnableAutoConfiguration, Spring Boot scans all auto-configuration classes listed in META-INF/spring.factories (2.7+ changed to META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports), and decides whether they take effect based on @Conditional annotations.

5. RocketMQ Deep Dive

This was the highlight of the interview — 9 consecutive questions, from basic scenarios to the complete logic of distributed locks for preventing duplicate consumption, digging very deep.

1. What business scenarios is MQ typically used for?

Reference answer:

ScenarioDescriptionExample
Async processingNon-core logic async, improving response timeSend SMS/email asynchronously after user registration
Traffic peak shavingBuffer burst traffic through MQ, protecting downstreamFlash sale orders enter queue first, then consumed gradually
Application decouplingUpstream and downstream communicate via MQ, reducing couplingOrder system sends message; inventory/logistics systems each consume
Delayed schedulingDelayed messages for scheduled tasksOrder auto-cancel after 30 minutes unpaid

2. What’s the difference between local async thread processing and using MQ after receiving data via API? Why choose MQ?

Reference answer:

ComparisonLocal Async ThreadMQ
ReliabilityTasks lost if process crashesPersistent storage, no loss
ScalabilityLimited single-machine scalingConsumers can scale horizontally
Flow controlThread pool full → reject or queueMQ naturally provides peak shaving
DecouplingProducer and consumer in same processFully decoupled, independently deployable
RetryMust implement retry logic yourselfMQ has built-in retry mechanism
MonitoringMust implement yourselfMQ provides console and metrics

Core reason to choose MQ: Reliability and decoupling. Local thread pool tasks are lost on application restart or crash, while MQ provides persistence and retry mechanisms ensuring messages are eventually consumed. MQ also naturally supports multiple consumers, multiple languages, and cross-system integration — far more scalable than local threads.

3. How to ensure messages are consumed successfully?

Reference answer: RocketMQ provides a comprehensive ACK mechanism:

  • Consumer processes successfully: return CONSUME_SUCCESS, Broker marks message as consumed
  • Consumer processing fails: return RECONSUME_LATER, Broker puts message into retry queue
  • Retry mechanism: 16 retries by default with increasing intervals (10s, 30s, 1min…2h)
  • Exceeds max retries: message enters dead letter queue (DLQ), requires manual handling

Key principle: Business logic must be idempotent, as retries may cause duplicate consumption.

4. Have you handled message loss scenarios? How did you solve them?

Reference answer: Message loss can occur at three stages:

Producer side:

  • Cause: Network jitter causes send failure without retry
  • Solution: Use synchronous send() instead of sendOneway(), with retry mechanism; or use transaction messages

Broker side:

  • Cause: Broker crash, messages not flushed to disk
  • Solution: flushDiskType=SYNC_FLUSH (sync flush); master-slave sync brokerRole=SYNC_MASTER

Consumer side:

  • Cause: ACK before business processing completes; or business fails after successful consumption
  • Solution: Return CONSUME_SUCCESS only after business processing completes; return RECONSUME_LATER on failure to trigger retry

In our project we used: synchronous send + sync flush + ACK after business completion — triple guarantee.

5. How to implement ordered messages?

Reference answer: RocketMQ supports two types of ordered messages:

Global ordered messages:

  • Only one MessageQueue under the Topic; all messages consumed in strict send order
  • Poor performance; rarely used

Partition ordered messages (commonly used):

  • Messages with the same business ID are sent to the same Queue, ensuring order for that business ID
  • Messages with different business IDs can be consumed in parallel

Implementation:

Key point: The producer uses MessageQueueSelector to route the same business ID to the same Queue; the consumer uses MessageListenerOrderly to ensure serial consumption within each Queue.

6. What should you watch out for with MQ broadcast mode consumption?

Reference answer: In broadcast mode (MessageModel.BROADCASTING), each message is consumed by every consumer instance.

Key considerations:

  • No retry mechanism: Failed consumption in broadcast mode won’t trigger retries, because each consumer consumes independently and the Broker can’t manage consumption progress uniformly
  • No progress persistence: Consumption progress is saved locally, not submitted to the Broker. Instance restarts may cause duplicate consumption or message loss
  • Suitable scenarios: Global events that all instances need to be aware of, like cache refresh notifications, config change notifications
  • Unsuitable scenarios: Scenarios requiring each message to be processed exactly once (like order processing)

7. What problems occur when multiple consumers consume the same message? How to solve?

Reference answer: In cluster mode (default), multiple consumer instances within the same ConsumerGroup will load-balance consumption — each message is consumed by only one instance, no duplication.

However, consumers in different ConsumerGroups consuming the same Topic will each consume every message once — this is expected behavior.

Potential issues:

  • Duplicate consumption: During consumer instance scaling, Rebalance may cause duplicate consumption
  • Consumption backlog: A slow instance causes its assigned Queues to accumulate

Solutions:

  • Idempotent consumption: Business logic ensures idempotency — duplicate consumption has no side effects
  • Distributed locks: Use Redis distributed locks to ensure the same business ID isn’t processed concurrently

6. Project Challenges

Finally, we discussed a few real project challenges. The interviewer wanted to see your problem-solving approach in actual scenarios.

1. What challenging and valuable problems have you encountered at work?

Reference answer: The most challenging was the unified parsing and vectorization of multi-format documents in the LIMS project.

The difficulty was that lab documents came in all formats — PDFs included both scanned and electronic versions, Word documents had embedded tables and images, and Excel files had multiple sheets and merged cells. Each format required a completely different parsing approach. Moreover, if table data was chunked as plain text, the semantics would break, causing poor RAG retrieval performance.

Solution:

  1. Defined a unified DocumentParser interface with each format implementing its own parser
  2. Parsed results unified into DocumentChunk structure (title + content + metadata + source)
  3. Special handling for table data: converted row-column relationships into natural language descriptions (e.g., “the value in row 3, column 2 is XX”) before vectorization
  4. Extracted images processed by multimodal model to generate description text, then vectorized
  5. Chunking strategy: split by chapter headings to maintain semantic integrity

Result: knowledge base retrieval accuracy improved from 60% to 85%, with noticeably improved user satisfaction.

2. Excel read/write and embedded image parsing — how are images stored and processed afterwards?

Reference answer: Parsing embedded images in Excel is indeed challenging — POI’s support for this is not very friendly.

Parsing process:

  1. Read Excel file with XSSFWorkbook
  2. Get all embedded images via getAllPictures()
  3. Associate images with cells: iterate XSSFDrawing, get anchor info (XSSFClientAnchor) to determine the image’s row/column position
  4. Write image data to file system or object storage (MinIO), return access URL
  5. Record the image-cell association in the database

Storage options:

OptionDescriptionUse Case
Local file systemSimple and direct, but doesn’t support distributedSingle-machine deployment
MinIOS3-compatible, self-hosted object storageIntranet environment
OSS/S3Cloud provider object storagePublic cloud environment

We used MinIO because the production environment is intranet-deployed with no internet access. Image URLs are stored in the database and associated with business data.

3. Why don’t customers just use forms to upload images instead of Excel import?

Reference answer: I asked the customer this too. The reasons are:

  1. Historical habits: The lab has always managed data in Excel and has accumulated many Excel templates. They don’t want to change their workflow
  2. Batch efficiency: A single test record may contain dozens of fields and over a dozen images. Filling them one by one in a form is too slow; Excel allows batch filling and one-click import
  3. Offline scenarios: Some lab areas have poor network; Excel can be filled offline and uploaded when back at the office
  4. Data exchange: Testing equipment exports data in Excel format, which needs to be imported directly — it’s impossible to have operators manually enter everything

So this isn’t a technology choice issue — it’s determined by the business scenario. As developers, we need to adapt to customers’ actual workflows, not make customers adapt to our system design.

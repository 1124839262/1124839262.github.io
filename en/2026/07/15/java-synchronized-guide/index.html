Preface

In Java concurrent programming, the synchronized keyword is the most fundamental and commonly used synchronization mechanism. It is both a high-frequency topic in interviews and the primary tool for solving thread-safety issues in real-world development. However, many people’s understanding only scratches the surface of “locking,” and they struggle to go deeper.

This article provides a comprehensive and in-depth analysis, covering usage, low-level principles, lock upgrade mechanisms, JVM implementation, and common interview questions.

1. Three Ways to Use synchronized

1.1 Modifying Instance Methods (Instance Lock)

The lock is on the current instance object (this). Multi-threaded calls on the same instance are mutually exclusive, while different instances do not affect each other.

1.2 Modifying Static Methods (Class Lock)

The lock is on the Class object of the current class (SynchronizedDemo.class). Regardless of how many instances exist, only one thread can execute this static method at any given time.

1.3 Modifying Code Blocks

Code blocks allow specifying any object as the lock, offering the most flexibility. Common usage:

Lock ObjectScope
thisCurrent instance, equivalent to modifying an instance method
Xxx.classClass object, equivalent to modifying a static method
Custom ObjectFine-grained control, avoids conflicts with external locks

1.4 Are Instance Locks and Class Locks Mutually Exclusive?

No, they are not. An instance lock locks the object, while a class lock locks the Class object. They are entirely different locks and will not block each other.

Two threads calling instanceMethod() and classMethod() respectively can execute simultaneously without blocking.

3. Lock Upgrade (Inflation) Process in Detail

Since JDK 1.6, synchronized has introduced biased locks and lightweight locks to optimize performance. The lock can upgrade from a lower level to a higher level, but cannot downgrade.

3.1 Biased Locking

Core idea: In most cases, locks not only have no contention but are also always acquired by the same thread multiple times. A biased lock records the thread ID in the Mark Word, so when that thread re-enters the synchronized block, it only needs to check if the thread ID matches, without any CAS operation.

Acquisition process:

  1. Check if the biased lock flag is 1 and the lock flag bits are 01
  2. If biased, check if the thread ID recorded in Mark Word is itself
  3. If yes, directly execute the synchronized code without any synchronization operations
  4. If no, try to set the thread ID to itself via CAS
  5. If CAS succeeds, execute the synchronized code; if it fails, enter biased lock revocation

Biased lock revocation: When another thread tries to acquire the biased lock, the biased lock is revoked and upgraded to a lightweight lock. Revocation must occur at a global Safe Point, pausing the thread that holds the biased lock.

Note: Starting from JDK 15, biased locks are disabled by default (because in high-concurrency scenarios, the cost of biased lock revocation is high).

3.2 Lightweight Locking

Core idea: When a biased lock is accessed by another thread, it upgrades to a lightweight lock. Lightweight locks use spinning to wait for the lock to be released, avoiding the overhead of thread blocking and waking.

Acquisition process:

  1. Create a Lock Record space in the current thread’s stack frame
  2. Copy the Mark Word into the Lock Record (called Displaced Mark Word)
  3. Use CAS to update the Mark Word to a pointer to the Lock Record
  4. If CAS succeeds, acquire the lock; if it fails, enter spinning

Spinning: When a thread fails to acquire a lightweight lock, it does not immediately block but loops and retries CAS to acquire the lock. Since JDK 1.6, the number of spin attempts is adaptive, dynamically determined by the JVM based on historical behavior.

Lightweight lock release: The Displaced Mark Word is replaced back into the Mark Word via CAS. If CAS fails, it means another thread is contending, and the lock is upgraded to a heavyweight lock.

3.3 Heavyweight Locking

When the lightweight lock has spun a certain number of times (or more than one thread is contending), it inflates to a heavyweight lock.

Heavyweight locks rely on the operating system’s Mutex Lock implementation, causing thread blocking and context switching with significant overhead. However, threads do not consume CPU while spinning.

3.4 Lock Upgrade Summary

Lock StateAdvantageDisadvantageSuitable Scenario
Biased LockAlmost no overheadHigh revocation costOnly one thread accesses
Lightweight LockAvoids thread blockingConsumes CPU spinningThreads alternate execution
Heavyweight LockNo CPU consumptionHigh blocking/waking overheadHigh concurrency contention

4. Characteristics of synchronized

4.1 Reentrancy

synchronized is a reentrant lock. The same thread can acquire the same lock multiple times without deadlocking itself.

The _recursions field in the Monitor records the reentry count, incrementing by 1 on each reentry and decrementing by 1 on exit. The lock is released when the count reaches zero.

4.2 Non-Interruptibility

When a thread is waiting for a synchronized lock, it cannot be interrupted. Calling Thread.interrupt() will not cause the waiting thread to throw an InterruptedException.

4.3 Non-Fairness

synchronized is a non-fair lock. Threads in the waiting queue are not guaranteed to acquire the lock in order; newly arrived threads may compete with waiting threads.

4.4 Automatic Release

synchronized automatically releases the lock when the synchronized code completes execution or throws an exception, which is one reason it is easier to use than Lock.

7. Usage Recommendations and Precautions

  1. Minimize synchronization scope: Only lock the code that needs synchronization, reducing lock hold time.
  2. Use private final objects as locks: Prevents external code from accidentally acquiring the lock and causing deadlocks.
  3. Avoid calling external methods within locks: May cause deadlocks or performance issues.
  4. Do not use String constants as locks in synchronized: The String constant pool may cause different code to unexpectedly share the same lock.
  5. Avoid nested locks: Nesting multiple locks can easily lead to deadlocks; always maintain a consistent lock ordering.
  6. Prefer concurrent collections: Such as ConcurrentHashMap, which have better internal concurrency optimizations.

8. Summary

After JDK 1.6, synchronized has undergone extensive optimization (biased locks, lightweight locks, adaptive spinning, lock elimination, lock coarsening, etc.), and performance is no longer a bottleneck. In daily development, if you don’t need ReentrantLock’s advanced features (interruptibility, timeout, multiple conditions), synchronized is a cleaner and safer choice.

Appendix: Frequently Asked Interview Questions and Answers

Q1: How is synchronized implemented at the low level?

Answer: synchronized relies on the JVM’s Monitor mechanism.

  • For code blocks: The bytecode uses monitorenter and monitorexit instructions. monitorenter attempts to acquire the object’s Monitor; if successful, _count increments by 1. monitorexit decrements _count by 1, and the lock is released when it reaches zero.
  • For methods: The ACC_SYNCHRONIZED flag is set in the method table, and the JVM implicitly calls monitorenter/monitorexit based on this flag.

The underlying data structure is ObjectMonitor (implemented in C++), containing fields such as _owner (lock holder thread), _EntryList (contention queue), and _WaitSet (wait queue).

Q2: Describe the lock upgrade process of synchronized.

Answer: The lock upgrade direction is one-way: Biased Lock → Lightweight Lock → Heavyweight Lock, with no downgrade.

  1. Biased Lock: Initial state. When only one thread accesses synchronized code, the Mark Word records that thread’s ID. Subsequent entries by the same thread are granted directly without any synchronization operations.
  2. Lightweight Lock: When another thread contends for the biased lock, it is revoked and upgraded to a lightweight lock. Threads try to acquire the lock via CAS, and spin-wait on failure.
  3. Heavyweight Lock: When spinning reaches a threshold or the number of contending threads exceeds a limit, the lock is upgraded to heavyweight. Threads that fail to acquire the lock are blocked and enter the OS’s Mutex wait queue, consuming no CPU.

Q3: Is synchronized reentrant? How is it implemented?

Answer: Yes, synchronized is a reentrant lock.

Implementation: The ObjectMonitor maintains _recursions (reentry count) and _owner (holder thread) fields. When a thread tries to acquire the lock, it first checks if _owner is itself. If so, _recursions increments by 1 and the thread is granted access directly. Each time the thread exits a synchronized block, _recursions decrements by 1. The lock is only truly released when it reaches zero.

Q5: Why must wait() and notify() be called inside synchronized?

Answer: Because wait() and notify() are based on the Monitor mechanism. When wait() is called, the thread releases the currently held Monitor and enters the WaitSet to wait. When notify() is called, a thread is woken from the WaitSet to enter the EntryList and compete for the lock.

If called outside synchronized, the current thread does not hold the Monitor, and an IllegalMonitorStateException will be thrown.

Q6: What is the difference between synchronized on static methods and instance methods?

Answer:

  • Modifying instance methods: The lock is on the current instance object (this). Different instances are not mutually exclusive.
  • Modifying static methods: The lock is on the class’s Class object (Xxx.class). There is only one Class object globally, so all instances share the same lock.

The two lock on different objects, so they are not mutually exclusive.

Q7: If two threads access two different synchronized methods on the same object simultaneously, will one block?

Answer: Yes. Because both instance synchronized methods lock on the same instance object (this). When Thread A enters method 1, it holds the this lock. Thread B, wanting to enter method 2, also needs the this lock, so it will be blocked.

Q8: What are the four necessary conditions for deadlock? How to avoid it?

Answer:

Four necessary conditions (all must be present):

  1. Mutual Exclusion: A resource can only be held by one thread at a time.
  2. Hold and Wait: A thread holds resources while requesting new ones.
  3. No Preemption: Resources held by a thread cannot be forcibly taken away.
  4. Circular Wait: Threads form a circular waiting chain.

Avoidance methods:

  • Maintain consistent lock ordering to avoid circular waiting.
  • Use tryLock with timeout (ReentrantLock).
  • Use tools like jstack to detect deadlocks.

Q9: Does synchronized guarantee visibility?

Answer: Yes. Under the JMM (Java Memory Model), synchronized ensures:

  • Before locking, the thread must re-read the latest values of shared variables from main memory.
  • Before unlocking, the thread must flush shared variables from working memory back to main memory.

This guarantees that modifications within a synchronized block are visible to other threads acquiring the same lock.

Q10: What are lock elimination and lock coarsening?

Answer: These are two JVM optimizations for synchronized:

  • Lock Elimination: The JIT compiler, through escape analysis, removes synchronization operations if the lock object in a synchronized block cannot be accessed by other threads (e.g., a local variable).
  • Lock Coarsening: If a series of consecutive operations repeatedly lock and unlock the same object (e.g., in a loop), the JVM expands the lock scope into a single lock/unlock operation, reducing synchronization overhead.

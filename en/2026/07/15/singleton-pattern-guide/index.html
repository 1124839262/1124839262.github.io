Preface

The Singleton Pattern is the simplest and most commonly used design pattern. It ensures that a class has only one instance and provides a global access point to that instance.

Although it seems simple, implementing the Singleton Pattern in a multi-threaded environment has many nuances. This article details 7 implementation methods, from the basic Eager Initialization to the elegant Enum approach, analyzing their pros, cons, and applicable scenarios.

1. What is the Singleton Pattern

1.1 Definition

The Singleton Pattern ensures that a class has only one instance and provides a global access point to retrieve that instance.

1.2 Use Cases

  • Configuration management classes (e.g., reading config files)
  • Database connection pools
  • Thread pools
  • Caches (e.g., Redis clients)
  • Loggers
  • Application state managers

1.3 Implementation Points

  1. Private constructor: Prevents external instantiation via new
  2. Hold the unique instance: Create and hold the instance within the class
  3. Provide global access method: Get the instance through a static method

  • Singleton that is certain to be used
  • Scenarios not sensitive to startup speed

4. Static Inner Class (Holder Pattern, Recommended)

4.1 Principle

  • JVM guarantees that class initialization is thread-safe (class loader mechanism)
  • Static inner class Holder is only loaded when first referenced
  • Achieves lazy loading without locking

Lazy loading✅ Supported
Implementation complexity⭐⭐ Relatively simple
Performance⭐⭐⭐⭐⭐ No locks, best performance

4.3 Pros

  • Achieves both lazy loading and thread safety
  • No performance overhead
  • Clean and elegant code

5. Enum (Most Recommended)

5.1 Why is it Best Practice?

Recommended by Joshua Bloch (author of “Effective Java”) because:

  1. Thread-safe: Enum instance creation is thread-safe by JVM
  2. Prevents reflection attacks: Reflection cannot create enum instances
  3. Prevents serialization attacks: Enum deserialization returns the same instance
  4. Clean code: One line implements singleton

11. Interview Questions

Q1: What are the implementation methods of Singleton Pattern? Which is best?

Answer: There are 7 common implementation methods:

  1. Eager: Created when class loads, thread-safe, no lazy loading
  2. Lazy (Sync Method): Lazy loading, but poor performance
  3. Double-Checked Locking (DCL): Lazy loading + thread-safe, requires volatile
  4. Static Inner Class: Lazy loading + thread-safe + high performance, most elegant lazy loading solution
  5. Enum: Simplest, safest, naturally prevents reflection and serialization attacks, recommended by Joshua Bloch
  6. Container: Manages multiple singletons, suitable for frameworks
  7. ThreadLocal: Thread-isolated “singleton”

Recommendations:

  • No lazy loading needed: Enum
  • Lazy loading needed: Static Inner Class

Q2: Why does Double-Checked Locking need volatile?

Answer: instance = new Singleton() is not an atomic operation and may be reordered as:

  1. Allocate memory space
  2. Initialize object
  3. Point reference to memory address

If reordered to 1 → 3 → 2, after Thread A executes 3, Thread B checks instance != null and returns an uninitialized object.

volatile prevents instruction reordering, ensuring the execution order 2 → 3.

Q3: Why can Static Inner Class achieve lazy loading?

Answer: Utilizes the JVM’s class loading mechanism:

  • Static inner class Holder is only loaded when first referenced
  • SingletonHolder.getInstance() first call triggers Holder class loading
  • INSTANCE is initialized during class loading, JVM guarantees thread safety during class initialization

This achieves lazy loading without locking, with optimal performance.

Q5: What are the pros and cons of Singleton Pattern?

Answer:

Pros:

  • Only one instance in memory, saves resources
  • Global access point, easy to use
  • Avoids repeated creation, improves performance

Cons:

  • Violates Single Responsibility Principle (class handles both business logic and creation management)
  • Hides dependencies between classes
  • Not conducive to unit testing (difficult to mock)
  • Difficult to extend (no longer singleton after inheritance)

Q6: How are Singleton Beans implemented in Spring?

Answer: Spring uses Container Singleton (Registry Pattern):

  • DefaultSingletonBeanRegistry uses ConcurrentHashMap to store singleton beans
  • Key: Bean name
  • Value: Bean instance

Spring’s singleton scope is at the Spring container level, not JVM level.

12. Summary

Although the Singleton Pattern is simple, there are many details to consider when implementing it in a multi-threaded environment:

  1. Eager: Simple and efficient, but no lazy loading
  2. DCL: Supports lazy loading, requires volatile to prevent instruction reordering
  3. Static Inner Class: Best practice for lazy loading, lock-free high performance
  4. Enum: Safest implementation, naturally prevents reflection and serialization attacks

Practical Development Recommendations:

  • Prefer Enum (Effective Java recommendation)
  • Use Static Inner Class when lazy loading is needed
  • Avoid synchronized method version of lazy loading (poor performance)
  • Pay attention to defense against reflection, serialization, and clone attacks

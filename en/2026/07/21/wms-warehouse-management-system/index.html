1. What is WMS

WMS (Warehouse Management System) is a software system that manages daily warehouse operations, covering the entire process of inbound, outbound, inventory, picking, counting, and location management.

The difference from ERP: ERP manages “accounts”; WMS manages “physical goods.” ERP tells you there are 100 items in stock. WMS tells you exactly which shelf, which location, which batch, and which pallet those 100 items are on.

2.2 Inbound Management

Inbound flow:

ASN Booking → Receiving → Quality Check → Putaway → Location Registration

StageFunctionKey Points
ASN BookingSupplier notifies delivery in advanceIntegrate with ERP purchase orders, generate ASN number
ReceivingScan/enter arriving goodsSupport barcode/PDA scanning, verify quantity
Quality CheckInspect goods qualityPass/fail routing, rejected items go to quarantine
PutawayPlace goods into specific locationsAuto-recommend locations based on putaway strategy, PDA confirmation
Location RegistrationUpdate system inventoryRecord item, quantity, batch, location, pallet number

2.3 Outbound Management

Outbound flow:

Order Receipt → Wave → Inventory Allocation → Picking → Checking → Packing → Shipping

StageFunctionKey Points
Order ReceiptPull outbound orders from ERP/OMSSupport batch import, real-time push
Wave ManagementMerge orders into waves by rulesGroup by carrier, zone, priority
Inventory AllocationLock inventory, assign pick locationsSupport FIFO/FEFO/specific batch
PickingRetrieve items from locations per pick listPDA scan confirmation to prevent errors
CheckingVerify picked items are correctBarcode scan, weight verification
PackagingPack and labelGenerate shipping labels, track packaging consumption
ShippingHand over to logisticsUpdate order status, notify ERP

This is the most critical WMS module and the most error-prone.

3.2 Concurrency Control for Inventory Deduction

This is the most classic technical challenge in WMS. When multiple outbound orders simultaneously deduct inventory from the same location, how to prevent overselling?

`java
// Solution 1: Database row lock (recommended)
@Transactional
public void allocateInventory(Long itemId, Long locationId, String lotNo, BigDecimal qty) {
// SELECT … FOR UPDATE locks the row
Inventory inventory = inventoryMapper.selectForUpdate(itemId, locationId, lotNo);
if (inventory.getAvailableQty().compareTo(qty) < 0) {
throw new BusinessException(“Insufficient inventory”);
}
// Deduct inventory
inventory.setLockedQty(inventory.getLockedQty().add(qty));
inventory.setAvailableQty(inventory.getQty().subtract(inventory.getLockedQty()));
inventoryMapper.updateById(inventory);

// Record log
insertInventoryLog(itemId, locationId, lotNo, qty.negate(), "Outbound", orderNo);

}
`

6. Key Considerations and Lessons Learned

6.1 Inventory Consistency

  • Inventory log table is append-only — every change must be recorded for traceability
  • Use database row locks (SELECT FOR UPDATE) for inventory deduction to prevent overselling
  • Never calculate inventory at the application layer — let the database handle it: SUM(qty) GROUP BY item_id
  • Periodically perform inventory reconciliation: physical quantity vs. system quantity, auto-generate adjustment orders for discrepancies

6.2 Batch Management

  • If you don’t need batch tracking, don’t force it — it doubles the complexity
  • Recommended batch number format: date + sequence + supplier code, e.g., 20260721-001-S001
  • Expiry management: scheduled task scans near-expiry/expired items, auto-freeze or transfer

6.3 Barcode and Coding Standards

  • Use GS1-128 or Code 128 barcodes uniformly — 1D barcodes are sufficient, don’t overcomplicate with 2D codes
  • Location code format: zone-rack-level-position, e.g., A-01-03-02
  • Pallet coding: LPN + date + sequence, e.g., LPN202607210001

6.4 Performance Optimization

  • Add indexes on inventory queries: (item_id, location_id, lot_no, status)
  • Partition inventory log table by month to avoid massive single tables
  • Report queries should use read replicas, never hit the primary database
  • PDA API response time must be under 500ms, or field operators will complain

6.5 Fault Tolerance and Exception Handling

  • Inbound order status flow: Pending → Receiving → Completed, allow rollback from “Receiving” to “Pending”
  • Outbound order status flow: Pending → Allocated → Picking → Checked → Shipped, each stage can roll back
  • Exception scenarios:
    • Receiving overage/shortage → generate discrepancy order, notify purchasing
    • Wrong pick → allow pick cancellation, inventory rollback
    • Missed shipment → support reshipment, record reason
    • Counting discrepancy → generate discrepancy order, adjust after approval

6.6 Permission Design

Interview Questions

1. What’s the difference between WMS and ERP inventory?

ERP manages book inventory — it only records items and quantities. WMS manages physical inventory — precise to each location, each batch, each pallet.

Example: ERP says “Item A has 100 in stock.” WMS says “Item A has 30 at location A01-03-02 (batch 20260721) and 70 at location B02-01-01 (batch 20260720).”

2. How does WMS prevent inventory overselling?

Core solution: database row lock SELECT FOR UPDATE. When deducting inventory, first lock the row, read current stock, check if sufficient, then deduct, commit transaction to release the lock.

For high-concurrency scenarios, use Redis distributed locks, but ensure Redis-DB consistency — generally, the database should be the source of truth.

3. How to design the inbound and outbound state machines?

Inbound: Pending → Receiving → Pending Putaway → Completed. Rollback from “Receiving” to “Pending” is allowed, but not from “Completed.”

Outbound: Pending → Allocated → Picking → Checked → Shipped. Rollback from “Picking” to “Allocated” is allowed, but not after “Shipped.”

4. Why is the inventory log table append-only?

For audit trail. Every inventory change can be traced: who, when, what operation, and the quantity before and after. If you modify the inventory table directly, issues become untraceable.

5. How to do cycle counting? How to handle discrepancies?

  1. Create a counting task (full / partial / dynamic)
  2. Freeze locations in the counting range
  3. Counters use PDA to scan each location and enter physical quantities
  4. System auto-compares discrepancies
  5. Discrepancy order goes through approval
  6. After approval, system auto-adjusts inventory and records the log

6. What picking strategies are there?

StrategyDescriptionUse Case
Order PickingOne person picks one entire orderSmall order volume
Wave PickingMerge multiple orders, pick by item summary, then sortLarge order volume
Zone PickingDifferent people handle different zones, relay-styleLarge warehouse
Pick-and-SortPick and sort by order simultaneouslyMedium order volume

7. How does WMS integrate with hardware?

  • PDA: Android, calls backend API via HTTP, or uses WebSocket for real-time task push
  • Barcode Printer: Backend generates barcode commands (ZPL language), sends via Socket to printer
  • Scale: Read weight via serial port, verify pick weight is within tolerance
  • PLC/Conveyor: Send sorting instructions via Modbus TCP

  • Microservice split: inbound, outbound, inventory, location, strategy each as a service
  • Inventory service is the core — other services operate inventory through it

  • Report queries use read replicas to avoid impacting primary database performance

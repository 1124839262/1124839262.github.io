Preface

In a single-machine application, we can use synchronized or ReentrantLock to ensure thread safety. However, in a distributed system, multiple service instances run in different JVMs, and these local locks are powerless — you need a distributed lock to coordinate resource access across different processes.

For example, in an e-commerce system, multiple service instances process orders simultaneously. Without a distributed lock, “overselling” can occur — two requests both read the inventory as 1, both assume the order can be placed, and ultimately two items are sold.

There are many distributed lock implementations on the market. This article will introduce 6 mainstream solutions one by one, covering their principles, usage, and applicable scenarios:

1. Redisson：Redis-Based Distributed Lock Swiss Army Knife

1.1 What Is It

Redisson is a Redis-based Java In-Memory Data Grid. It is not just a Redis client — it also provides a series of advanced features such as distributed locks, distributed collections, and distributed objects.

In terms of distributed locks, Redisson implements the java.util.concurrent.locks.Lock interface, making it almost identical to using local locks.

1.2 Core Mechanism：Watchdog

The most outstanding design of Redisson’s distributed lock is the Watchdog automatic renewal mechanism：

This mechanism solves the classic problem of “the business hasn’t finished executing but the lock has already expired.” You don’t need to manually set a lock timeout — Redisson renews it automatically.

2.1 What Is It

Lock4j is a declarative distributed lock framework based on Spring AOP, using Redisson as the default underlying implementation (other implementations can also be configured). Its core philosophy is: lock through annotations, zero intrusion into business code.

3. ShedLock：Distributed Scheduled Task Lock

3.1 What Is It

ShedLock is a lock framework specifically designed for distributed scheduled tasks. Its positioning is very clear: ensure that the same scheduled task runs on only one instance at any given time.

This is not a general-purpose distributed lock, but specifically solves the problem of “scheduled tasks executing repeatedly when deployed on multiple instances.”

3.2 Core Mechanism

ShedLock writes a record to the storage layer (database/Redis/ZooKeeper) before task execution, and deletes or updates the record after execution. When other instances attempt to execute the task, they skip it if the record already exists.

5. Curator InterProcessMutex：ZooKeeper-Based Distributed Lock

5.1 What Is It

Apache Curator is a Java client framework for ZooKeeper, and InterProcessMutex is the distributed reentrant lock implementation it provides.

ZooKeeper is based on the CP model (Consistency + Partition tolerance), making it naturally suitable for distributed locks.

5.2 Core Mechanism：Ephemeral Sequential Nodes

The implementation principle of ZooKeeper distributed locks：

  1. All clients create ephemeral sequential nodes under /locks
  2. The node with the smallest sequence number holds the lock
  3. Each client watches the preceding node
  4. When the preceding node is released (client disconnects or actively deletes), the next node acquires the lock
  5. When a client crashes, the ephemeral node is automatically deleted — no deadlock

Selection Guide

Important Notes

  1. Lock granularity should be reasonable：Don’t lock the entire method; lock at the resource level (e.g., by order ID)
  2. Set reasonable timeout values：Prevent deadlocks, but not so short that the lock is released before business logic completes
  3. Validate before releasing locks：Ensure you only release locks you own, avoid accidentally deleting other clients’ locks
  4. Redisson’s Watchdog only works for lock() without leaseTime：If you specify an expiration time, the Watchdog won’t renew
  5. Database locks require cleanup of expired locks：Otherwise the table will grow indefinitely and may cause deadlocks
  6. ZooKeeper locks require attention to Session timeout：Network jitter can cause ephemeral nodes to be deleted, unexpectedly releasing locks

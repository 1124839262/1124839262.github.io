1. JVM Overview

1.1 What is JVM

JVM (Java Virtual Machine) is the core of the Java ecosystem. It is an abstract computer responsible for executing Java bytecode. The reason Java achieves “write once, run anywhere” is precisely the JVM — developers compile .class bytecode files that can run on any platform with a JVM installed.

2.2 Heap Structure

The heap is the primary area managed by GC, divided into Young Generation and Old Generation:

┌──────────────────────────────────────────┐ │ Heap │ │ ┌─────────────────────┐ │ │ │ Young Generation │ Eden:S0:S1 │ │ │ ┌────┬─────┬─────┐ │ = 8:1:1 │ │ │ │Eden│ S0 │ S1 │ │ │ │ │ └────┴─────┴─────┘ │ │ │ └─────────────────────┘ │ │ ┌─────────────────────┐ │ │ │ Old Generation │ │ │ └─────────────────────┘ │ └──────────────────────────────────────────┘

  • Young Generation: Newly created objects are allocated here first, cleaned by Minor GC
    • Eden Space: Most objects are born here
    • Survivor Spaces (S0/S1): Objects that survive one Minor GC are moved from Eden to Survivor
  • Old Generation: Objects that survive multiple GCs in the Survivor area are promoted to Old Generation, cleaned by Major GC / Full GC

2.3 Metaspace

Before JDK 8, it was called PermGen; after JDK 8, it was replaced by Metaspace:

AdvantageSimpleAvoids OOM, auto-expansion

3.2 Class Loaders and Parent Delegation Model

mermaid graph TD A[Bootstrap ClassLoader] --> B[Extension ClassLoader] B --> C[Application ClassLoader] C --> D[Custom ClassLoader]

Class LoaderLoading Scope
Bootstrap ClassLoaderCore class libraries under <JAVA_HOME>/lib (rt.jar, etc.)
Extension ClassLoaderExtension libraries under <JAVA_HOME>/lib/ext
Application ClassLoaderClasses on the user classpath

Parent Delegation Model: When a class loader receives a class loading request, it does not load it immediately. Instead, it delegates the request to its parent loader, all the way up to the Bootstrap ClassLoader. If the parent loader cannot find the class, the child loader loads it itself.

Benefits:

  • Avoids duplicate class loading
  • Prevents core class libraries from being tampered with (e.g., a custom java.lang.String will not be loaded)

4. Garbage Collection (GC)

4.1 How to Determine if an Object is Recyclable

AlgorithmPrinciplePros/Cons
Reference CountingEach object maintains a reference count; reclaimed when count reaches 0Simple and efficient, but cannot handle circular references
Reachability AnalysisStarting from GC Roots, traverse reference chains; unreachable objects are recyclableUsed by JVM, accurate but requires STW

GC Roots include:

  • Objects referenced in the VM stack
  • Objects referenced by static properties in the method area
  • Objects referenced by constants in the method area
  • Objects referenced in the native method stack
  • Objects held by synchronization locks

4.2 Garbage Collection Algorithms

AlgorithmProcessProsCons
Mark-SweepMark live objects → Sweep unmarked objectsSimpleMemory fragmentation
Mark-CompactMark live objects → Move live objects to one end → Clear beyond boundaryNo fragmentationHigh moving cost
CopyingDivide memory into two halves, use one at a time; copy live objects to the other half during GCNo fragmentation, efficientLow memory utilization (50%)
Generational CollectionCopying for Young Gen, Mark-Compact/Mark-Sweep for Old GenBest overall efficiencyComplex implementation

Interview Questions

1. Describe the JVM memory model

JVM memory is divided into thread-shared and thread-private areas.

Thread-shared: Heap (stores object instances and arrays) and Method Area (Metaspace since JDK 8, stores class info, constants, static variables).

Thread-private: VM Stack (stack frames created for each method execution, storing local variables, operand stack, etc.), Native Stack (serves Native methods), PC Register (records the current thread’s execution position).

2. What is the structure of the heap?

The heap is divided into Young Generation and Old Generation. The Young Generation is further divided into 1 Eden space and 2 Survivor spaces (S0, S1), with a default ratio of 8:1:1. New objects are first allocated in Eden. After surviving a Minor GC, objects move to a Survivor space. After surviving multiple GCs in the Survivor spaces, objects are promoted to the Old Generation. Large objects may be allocated directly in the Old Generation.

3. What is the Parent Delegation Model? Why is it designed this way?

The Parent Delegation Model means: when a class loader receives a loading request, it first delegates to its parent loader. Only if the parent loader cannot find the class does the child loader attempt to load it.

Design reasons: First, it avoids duplicate loading — the same class won’t be loaded multiple times by different loaders. Second, it’s for security — it prevents core class libraries from being tampered with. For example, a custom java.lang.String will not be loaded, ensuring the security of core APIs.

4. How to determine if an object can be garbage collected?

The JVM uses the reachability analysis algorithm. Starting from GC Roots, it traverses reference chains downward. If an object has no reference chain connecting it to any GC Root, it is marked as recyclable.

GC Roots include: objects referenced in the VM stack, objects referenced by static properties, objects referenced by constants, objects referenced in the native method stack, and objects held by synchronization locks.

5. What are the common GC algorithms? What are their pros and cons?

AlgorithmProsCons
Mark-SweepSimpleCauses memory fragmentation
Mark-CompactNo fragmentationHigh cost of moving objects
CopyingNo fragmentation, efficientWastes half the memory
Generational CollectionBest overall efficiencyComplex implementation

The JVM actually uses generational collection: Young Generation objects die young and are handled with copying; Old Generation objects have high survival rates and are handled with mark-sweep or mark-compact.

7. What are Minor GC, Major GC, and Full GC?

  • Minor GC: Occurs in the Young Generation, triggered when Eden is full. High frequency, fast.
  • Major GC: Occurs in the Old Generation, triggered when Old Generation space is insufficient. Usually accompanied by a Minor GC.
  • Full GC: Cleans the entire heap and Method Area. Long STW time, should be avoided whenever possible.

8. What conditions trigger a Full GC?

  • Calling System.gc() (suggested but not guaranteed to execute immediately)
  • Insufficient Old Generation space
  • Insufficient Method Area (Metaspace) space
  • During Minor GC, Survivor space cannot hold objects and Old Generation cannot either (promotion failure)
  • Concurrent Mode Failure during CMS GC

9. How to troubleshoot an online OOM?

  1. Check logs: First look at GC logs and application logs to determine if it’s a heap overflow or Metaspace overflow
  2. Dump file: If -XX:+HeapDumpOnOutOfMemoryError is configured, directly analyze the dump file
  3. Generate dump with jmap: jmap -dump:format=b,file=heap.hprof
  4. Analyze with MAT: Open the dump file with Eclipse MAT, check Dominator Tree and Leak Suspects
  5. Locate large objects: Find the objects consuming the most memory, analyze reference chains, and locate the code position

10. How do you typically approach JVM tuning?

  1. Choose collector: Select the appropriate GC based on business scenarios (G1/CMS for response priority, Parallel for throughput priority)
  2. Set heap size: Set -Xms and -Xmx to the same value to avoid performance loss from dynamic resizing
  3. Set Young Generation: Adjust -Xmn or -XX:NewRatio based on object lifecycle
  4. Enable GC logging: -XX:+PrintGCDetails -XX:+PrintGCDateStamps
  5. Analyze GC logs: Focus on GC frequency, pause time, and collection effectiveness, then tune iteratively
  6. Verify with stress testing: Use stress testing tools to verify tuning results, iterate repeatedly

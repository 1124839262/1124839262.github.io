Preface

In a monolithic application, transaction management is relatively simple — Spring’s @Transactional annotation can handle everything. But when you split a monolith into microservices, a single local transaction becomes a distributed transaction spanning multiple services and databases, and things get complicated.

For example, in an e-commerce order flow, the order service needs to create an order, the stock service needs to deduct inventory, and the account service needs to deduct the balance. These three operations run on three different servers and operate on three different databases. If the stock deduction succeeds but the account deduction fails, traditional transaction mechanisms cannot guarantee data consistency.

This article will guide you from theory to practice, covering distributed transaction solutions comprehensively, with a focus on Seata’s usage scenarios and hands-on approach.

1. Distributed Transaction Fundamentals

1.1 CAP Theorem

In distributed systems, the CAP theorem states that a distributed system can satisfy at most two of the following three guarantees simultaneously:

PropertyMeaningDescription
ConsistencyAll nodes see the same data at the same timeAfter a write, all reads return the latest value
AvailabilityEvery request receives a non-error responseThe system remains available without returning errors or timeouts
Partition ToleranceThe system continues to operate despite network partitionsCommunication failures between nodes do not affect overall service

Since network partitions are inevitable (P must be satisfied), distributed systems can only choose between CP and AP. Distributed transactions naturally lean toward CP, but trade-offs are made based on scenarios.

1.2 BASE Theory

BASE theory extends the AP side of CAP, advocating:

  • Basically Available: The system allows partial loss of availability during failures, while core functionality remains available
  • Soft state: The system allows intermediate states of data that do not affect overall system availability
  • Eventually consistent: All data replicas in the system will eventually reach a consistent state over time

BASE theory is the foundation of “flexible transactions” in distributed transaction design — it does not demand real-time consistency but guarantees eventual consistency.

2. Classic Distributed Transaction Solutions

2.1 Two-Phase Commit (2PC / XA)

2PC is the most classic distributed transaction protocol. It introduces a Transaction Coordinator to coordinate commits across multiple participants.

Execution Flow:

Advantages:

  • Simple principle, supported by mainstream databases (MySQL since 5.5 supports XA)
  • Strives for strong consistency

Disadvantages:

  • Single point of failure: If the coordinator crashes, all participants are blocked
  • Synchronous blocking: Resources are locked after Phase 1 and cannot be released until Phase 2 completes
  • Data inconsistency risk: If some participants receive Commit while others do not during Phase 2, data inconsistency occurs

2.2 TCC (Try-Confirm-Cancel)

TCC is a compensation-based transaction approach that splits a transaction into three phases:

PhaseMeaningOperation
TryAttempt executionPerform business checks and reserve resources
ConfirmConfirm commitExecute the actual business using resources reserved in Try
CancelCancel rollbackRelease resources reserved in the Try phase

Example: “Buying a bottle of water for $1”:

Advantages:

  • Lock-free, high performance
  • Not dependent on database transactions, can be used for cross-system calls

Disadvantages:

  • High business invasiveness, requires implementing Try/Confirm/Cancel for each operation
  • Must handle idempotency, empty rollback, and suspension issues

2.3 Local Message Table

The core idea of the local message table approach is to split a distributed transaction into multiple local transactions and guarantee eventual consistency through a message table.

Advantages:

  • Simple implementation, no external dependencies
  • Supports eventual consistency

Disadvantages:

  • Message table couples with business logic
  • Requires additional scheduled tasks for retry handling

2.4 MQ Transaction Messages

RocketMQ has supported transaction messages since version 4.3, essentially embedding the local message table logic into the MQ itself.

Advantages:

  • Low business invasiveness
  • Native RocketMQ support, high reliability

Disadvantages:

  • Only supports RocketMQ
  • Requires implementing the check-back interface

3. Seata: One-Stop Distributed Transaction Solution

3.1 What is Seata

Seata (Simple Extensible Autonomous Transaction Architecture) is an open-source distributed transaction solution from Alibaba, providing high-performance, low-intrusion distributed transaction services. Its predecessor was Alibaba’s internal GTS (Global Transaction Service). It was open-sourced in 2019 and entered the Apache Incubator, graduating as an Apache Top-Level Project in 2023.

3.2 Seata Core Architecture

Seata uses a three-layer architecture:

ComponentRoleResponsibility
TC (Transaction Coordinator)CoordinatorIndependently deployed server, maintains global transaction state, coordinates branch transaction commit or rollback
TM (Transaction Manager)ManagerEmbedded in business applications, defines global transaction boundaries, initiates global transaction begin/commit/rollback
RM (Resource Manager)ResourceEmbedded in business applications, manages resources involved in branch transactions, registers with TC and reports status

3.3 Seata’s Four Transaction Modes

Seata provides four transaction modes, each suited for different scenarios:

3.3.1 AT Mode (Automatic Transaction)

AT mode is Seata’s default and most widely used mode. It is based on the two-phase commit protocol but with significant optimizations, achieving automatic rollback through auto-generated reverse SQL.

Execution Flow:

Core Mechanisms:

  • undo_log table: Each business database requires an undo_log table to record before-image and after-image of data changes
  • Global lock: Seata acquires a global lock before commit to prevent other transactions from modifying the same data
  • Read-write isolation: Default read committed, write operations achieve isolation through global locks

Advantages:

  • Zero business code intrusion, just add @GlobalTransactional annotation
  • Auto-generated rollback logs, no need to manually write compensation logic

Disadvantages:

  • Only supports relational databases (via JDBC)
  • Requires creating undo_log table in each business database
  • Default isolation level is read uncommitted; caution needed in high-concurrency scenarios

3.3.2 TCC Mode

Seata’s TCC mode is consistent with classic TCC but provides framework-level support to help developers handle idempotency, empty rollback, and suspension issues.

Key Seata TCC Annotations:

useTCCFence = true enables Seata’s TCC anti-suspension mechanism, which automatically handles:

  • Empty rollback: Cancel is called when Try was never executed, automatically ignored
  • Suspension: Cancel arrives before Try, Try is rejected
  • Idempotency: Duplicate Try/Confirm/Cancel calls are automatically deduplicated

Advantages:

  • High performance, lock-free
  • Not dependent on databases, can call third-party APIs

Disadvantages:

  • High business invasiveness, requires implementing three methods
  • Complex compensation logic

3.3.3 Saga Mode

Saga mode is a long transaction solution that splits a long transaction into multiple ordered local transactions, each with a corresponding compensation operation.

Execution Flow:

Saga Implementation Approaches:

  • Choreography: Each service listens for events and decides the next step
  • Orchestration: A Saga coordinator centrally schedules operations (Seata uses this approach)

Seata Saga Features:

  • Supports JSON or Java-defined state machines
  • Supports service orchestration, suitable for complex long flows
  • Asynchronous execution, lock-free, high performance

Advantages:

  • Lock-free, suitable for long transactions
  • Supports non-database operations
  • Asynchronous execution, high throughput

Disadvantages:

  • Eventually consistent, cannot guarantee real-time consistency
  • Complex compensation logic, requires careful design

3.3.4 XA Mode

Seata’s XA mode is based on the database’s XA protocol, pursuing strong consistency.

Execution Flow:

Advantages:

  • Strong consistency, no data loss
  • Native database support, high reliability

Disadvantages:

  • Requires database XA protocol support
  • Synchronous blocking, lower performance
  • Occupies database connections for longer periods

4. Seata Mode Selection Guide

Based on different business scenarios, the following selection strategy is recommended:

Typical Scenario Recommendations

ScenarioRecommended ModeReason
E-commerce order (create order + deduct stock + deduct balance)ATAll DB operations, short transaction, zero intrusion
High-concurrency flash saleTCCHigh performance, lock-free, customizable optimization
Cross-system calls (including third-party APIs)TCCAT cannot handle non-DB operations
Cross-border payment / logistics schedulingSagaLong flow, tolerates eventual consistency
Financial core transactionsXAStrong consistency required
Legacy system migration (code cannot be modified)ATZero intrusion, just add annotation
Scheduled task batch processingSagaAllows long-running, asynchronous execution

6. Seata Production Considerations

6.1 High Availability Deployment

  • Deploy at least 2 Seata Server instances, using Nacos for service discovery
  • Use MySQL master-slave for transaction log storage to avoid single point of failure
  • Use Nacos cluster for config center and registry center

6.2 Performance Optimization Tips

OptimizationRecommendation
Regular undo_log cleanupConfigure scheduled tasks to clean up expired undo_log records
Set appropriate transaction timeoutDefault 60s, adjust upward for long transactions
Avoid large transactionsKeep branch transaction count within a single global transaction manageable
Database connection poolIncrease pool size appropriately to avoid connection waiting
Async commitConsider async processing for non-critical paths to reduce transaction scope

7. Summary

This article started from the fundamentals of distributed transactions, covering CAP theorem, BASE theory, and four classic distributed transaction solutions: 2PC/XA, TCC, Local Message Table, and MQ Transaction Messages. Then it focused on Seata’s four modes — AT, TCC, Saga, and XA — along with their applicable scenarios, and provided complete Spring Boot integration code.

Quick Selection Reference:

  • Most scenarios → AT Mode, zero intrusion, highest development efficiency
  • High concurrency or third-party APIs → TCC Mode
  • Long flows, cross-system → Saga Mode
  • Financial-grade strong consistency → XA Mode

There is no silver bullet for distributed transactions — each solution has its applicable scenarios and limitations. In real projects, the most important thing is to choose the most suitable solution based on business requirements, rather than pursuing the most “advanced” technology.

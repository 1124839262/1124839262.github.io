Preface

In the era of monolithic applications, method calls were simply Class A calling Class B’s methods, with no network overhead whatsoever. However, as businesses grow, monolithic architectures are gradually being split into microservices, and what were once local method calls become cross-process, cross-network remote calls.

At this point, you need a remote invocation framework to handle network communication, serialization, load balancing, service governance, and a host of other issues. Dubbo is precisely such a high-performance Java RPC framework built for this purpose.

This article will take you from zero to a comprehensive understanding of Dubbo’s use cases, usage methods, and a comparison with Spring Cloud Feign, helping you understand why Dubbo is a serious contender in microservice architecture.

1. What is Dubbo

Apache Dubbo is a high-performance, lightweight Java RPC (Remote Procedure Call) framework open-sourced by Alibaba. It provides three core capabilities:

  • Remote Method Invocation: Call remote services as if they were local methods
  • Intelligent Fault Tolerance and Load Balancing: Built-in multiple load balancing strategies and fault tolerance mechanisms
  • Automatic Service Registration and Discovery: Service providers and consumers automatically register and discover each other

Simply put, Dubbo solves the problem of “how services communicate with each other” in a microservice architecture.

2. Why Do You Need Dubbo

2.1 Communication Challenges from Monolith to Microservices

Imagine you have an e-commerce system split from a monolith into three microservices: product service, order service, and user service. When the order service needs to query product information, you face the following challenges:

Problems with using raw HTTP:

  1. Code Coupling: You need to manually concatenate URLs, manage HTTP connections, and parse responses
  2. Inefficient Serialization: JSON serialization is slow and carries redundant data
  3. Complex Connection Management: Short-lived connections have high overhead per request; long-lived connections require manual connection pool management
  4. No Service Governance: Load balancing, circuit breaking, and rate limiting must all be implemented manually
  5. No Service Discovery: Hardcoded service addresses require restarts when IPs change

How Dubbo solves this:

3. Dubbo Core Concepts and Architecture

3.1 Overall Architecture

Five Core Roles:

RoleDescription
ProviderService provider, the application that exposes services
ConsumerService consumer, the application that invokes remote services
RegistryRegistry center, responsible for service address registration and discovery
MonitorMonitoring center, collects call counts and response times
ContainerService runtime container

3.2 Invocation Flow

  1. When a Provider starts, it registers its service address with the Registry
  2. When a Consumer starts, it subscribes to the services it needs from the Registry
  3. The Registry returns the Provider address list to the Consumer; if there are changes, the Registry pushes the updated data
  4. The Consumer selects one Provider from the address list based on the load balancing algorithm and makes a direct call
  5. Both Provider and Consumer periodically send call statistics to the Monitor

4. Dubbo Use Cases

4.1 High-Performance RPC Calls Between Microservices

This is Dubbo’s core use case. In a microservice architecture, inter-service calls are frequent and require high performance and low latency.

Typical Scenario: In an e-commerce platform, the order service calls the product service, inventory service, user service, and payment service. Each order placement may involve 5-10 remote calls. If all went through HTTP + JSON, the latency would be extremely high.

Why Dubbo Fits:

  • Long-lived connection reuse, avoiding frequent handshakes
  • Binary serialization, smaller payload and faster processing
  • Client-side load balancing, eliminating one network hop

Key Takeaways

QuestionAnswer
Does the Consumer need an implementation class?No. Dubbo generates a “fake” implementation via dynamic proxy
What is the proxy object?An object implementing the ProductService interface, internally holding a Netty client connection
Why must the interface be shared?Provider and Consumer must agree on “what the interface looks like,” otherwise serialization/deserialization will fail
Is the proxy object a singleton?Yes, @DubboReference injects a singleton proxy by default
What if the Provider goes down?The proxy handles it according to the fault tolerance strategy (failover/failfast, etc.) and auto-detects node removal from the registry

Analogy: Restaurant Ordering

Think of Dubbo’s invocation model as ordering at a restaurant:

  • Menu (Interface): Both you and the kitchen have a copy; everyone follows the menu
  • Waiter (Proxy): Relays your order and brings the food; you don’t need to go into the kitchen
  • Kitchen (Provider): Where the actual work happens

You don’t need to know where the kitchen is or who the chef is — just order from the menu, and the waiter takes care of everything. That’s Dubbo’s RPC proxy mechanism in a nutshell.

6. Dubbo vs Feign (OpenFeign)

This is one of the most common questions from developers. Feign (now OpenFeign) is a declarative HTTP client in the Spring Cloud ecosystem, while Dubbo is an RPC framework. Although both are used for inter-service communication, their design philosophies and applicable scenarios differ.

6.1 Core Differences

DimensionDubboFeign (OpenFeign)
Communication ProtocolCustom dubbo protocol / Triple (binary fixed-length protocol)HTTP/1.1 + JSON (text protocol)
TransportLong connection + NIO multiplexingShort connection (HTTP 1.1) or connection pool
SerializationHessian2, Protobuf, Kryo (binary)Default JSON (text)
RegistryZooKeeper, Nacos, Redis, etcd, etc.Eureka, Nacos, Consul, etc.
Invocation StyleInterface proxy (interface-oriented)Interface proxy + annotations (HTTP-oriented)
Load BalancingClient-side built-in, multiple strategiesClient-side Ribbon/LoadBalancer
Fault Tolerance6 built-in cluster fault tolerance strategiesRequires Hystrix/Sentinel/Resilience4j
Service GovernanceBuilt-in routing, degradation, circuit breaking, weightingRelies on Spring Cloud ecosystem components
Cross-LanguageDubbo 3.x Triple supportsNative support (HTTP protocol)
Browser AccessibleNo (requires gateway conversion)Yes
PerformanceHigh (binary fixed-length protocol)Medium (HTTP text protocol)
Learning CurveMedium (need to understand RPC concepts)Low (similar to Spring MVC annotations)

6.2 Performance Comparison

Here is a simple performance comparison (calling the same simple interface returning a fixed object):

Why is Dubbo faster? The key reasons are:

  1. Protocol Level: The Dubbo protocol is fixed-length and binary, making parsing fast; HTTP is a text protocol with significant header overhead
  2. Serialization: Hessian2 is compact binary serialization; JSON is text serialization, with a 3-5x payload size difference
  3. Connection Model: Dubbo reuses long-lived connections, reducing TCP handshake overhead; HTTP 1.1, even with Keep-Alive, still has connection management overhead
  4. Network Model: Dubbo uses Netty NIO (non-blocking I/O); Feign’s underlying Servlet container uses blocking I/O by default

6.4 Coding Style Comparison

From a coding perspective, both achieve declarative invocation, but Dubbo is closer to the “interface-oriented programming” philosophy — the interface definition is the contract, and both provider and consumer adhere to it. Feign is closer to HTTP’s RESTful style, with the interface definition containing HTTP method and path information.

Final Thoughts

Dubbo is not a “trendy” framework, but it is a battle-tested, mature, and reliable RPC framework. Since its open-sourcing in 2011, Dubbo has supported tens of thousands of services and hundreds of thousands of instances within Alibaba — a scale of validation that few other frameworks can match.

If your project is a pure Java microservice architecture with requirements for performance and governance, Dubbo is a technical choice well worth considering. If you’re still using Spring Cloud Feign for internal service calls, try benchmarking Dubbo’s performance — you might be pleasantly surprised.
